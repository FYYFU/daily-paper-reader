## 摘要
LLM编码代理越来越多地被部署在真实的生产基础设施上自主行动。它们执行Shell命令、修改仓库、调用操作API。然而，完成任务并不足以保证安全。一个错误的操作可能导致严重后果。现有的代理基准主要强调任务完成，对于代理在良性但模糊的指令下的行为表现尚未明确。我们提出了UnderSpecBench，一个用于测量编码代理（如Claude Code、Codex和OpenCode）在DevOps任务中操作边界违规的基准。UnderSpecBench包含69个任务族，这些任务族基于有记录的事件、CVE或工具行为，并分布在四个DevOps能力域和九个操作控制面上。为了将模糊性与任务难度分离，每个任务保持相同的环境和真实安全操作，同时沿着三个维度（意图清晰度、目标确定性和爆炸半径）改变指令。由此产生的2,208个提示变体通过确定性、基于副作用的神谕（分离安全成功、错误目标和超范围结果）进行评估；非操作运行进一步分类为澄清、拒绝或推迟。在使用OpenCode、Claude Code和Codex的五种代理×模型配置中，评估结果表明模糊性主要不是导致代理失败，而是导致它们猜测。55.8-67.8%的运行违反了至少一个边界。目标不明确性显著降低了操作质量，而爆炸半径线索几乎没有减少操作倾向。这些发现表明，以完成为中心的评估可能夸大了安全自主性，并推动在模型、接口和系统层进行缓解措施。

## Abstract
LLM coding agents are increasingly deployed to act autonomously on real production infrastructure. They execute shell commands, modify repositories, and call operational APIs. However, completing a task is not sufficient for safety. A wrong action can cause severe consequences. Existing agent benchmarks largely emphasize task completion, leaving open how agents behave under benign but underspecified instructions.   We present UnderSpecBench, a benchmark for measuring action-boundary violations in coding agents (i.e., Claude Code, Codex, and OpenCode) on DevOps tasks. UnderSpecBench includes 69 task families grounded in documented incidents, CVEs, or tool behavior and organized across four DevOps capability domains and nine operational control surfaces. To isolate underspecification from task difficulty, each task keeps the same environment and ground-truth safe action while varying the instruction along three axes: intent clarity, target certainty, and blast radius. The resulting 2,208 prompt variants are evaluated with deterministic, side-effect-based oracles that separate Safe Success, Wrong Target, and OverScope outcomes; non-action runs are further classified as clarification, refusal, or deferment.   Across five agent x model configurations using OpenCode, Claude Code, and Codex, the evaluation results show that underspecification does not mainly make agents fail; it makes them guess. 55.8-67.8% of runs violate at least one boundary. Target underspecification sharply degrades action quality, while blast-radius cues barely reduce action propensity. These findings show that completion-centric evaluation can overstate safe autonomy and motivate mitigations at the model, harness, and system layer.

- **核心问题**：指令不明确（缺乏对意图、目标、范围的清晰说明）是否导致代理做出猜测性、越界的行动？现有以完成为中心的评分方式是否高估了安全自主性？
- **研究动机**：真实事件（如PocketOS事故、GitLab数据库删除等）表明，缺失目标或范围边界是操作事故的主因。因此需要一种新的评估方法，专门测量代理在模糊指令下是否尊重安全边界。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- 将指令模糊性作为受控实验变量，通过三种独立的模糊维度（意图清晰度、目标确定性、爆炸半径）构造指令变体，同时固定环境和真实安全操作，从而隔离模糊性对代理行为的影响。
- 使用**确定性、基于副作用的神谕（oracle）** 对每个运行进行细粒度评分，区分安全成功、错误目标、越界，而非简单的完成/未完成。

### 关键技术细节
1. **任务族构建**：基于真实事件、CVE或工具文档，提取69个DevOps任务族，覆盖4个能力域（变更接收与源码配置、构建与验证、发布与制品供应链、运行时与可靠性操作）和9个操作控制面。
2. **模糊性三轴**：
   - **意图清晰度（S）**：4级（S0完全明确→S3高度模糊），描述操作意图和环境的明确程度。
   - **目标确定性（B）**：4级（B0唯一标识→B3模糊引用），描述指令中目标对象的唯一性。
   - **爆炸半径（R）**：2级（R0低影响→R1高影响），描述操作后果的范围和不可逆性。
   - 每个任务族有4×4×2=32个指令变体，总计69×32=2208个不同提示。
3. **测试神谕**：每个任务有手写的确定性规范，定义**预期集合**（应修改的目标对象和范围）和**超范围集合**（必须保持不变的资源）。通过对比运行前后状态差异（文件diff、命令日志、服务副作用）自动判定三个布尔信号：
   - `INTENDED`：是否执行了真实安全操作
   - `WRONG_TARGET`：是否修改了错误对象
   - `OVER_SCOPE`：是否超出最小范围（即使目标正确但命令过宽）
4. **非操作分类**：对未采取行动的运行，使用LLM judge（DeepSeek-v4-flash）进一步分为**ASK**（询问澄清）、**REFUSE**（拒绝）、**DEFER**（分析/计划但未行动）。

- **基准**：UnderSpecBench本身为提出的基准，没有直接与其他现有基准对比，而是通过自身指标衡量行为。

  5. Codex + Codex-5.1-mini（同模型不同支架）

## 6. 论文的主要结论与发现

1. **动作倾向**：模糊性下代理仍然偏向行动，而非谨慎确认。行动率随S和B下降，但从不达到完全弃权；爆炸半径（R）几乎不影响行动率（R0:65.5% vs R1:64.0%）。
2. **动作质量**：目标确定性（B）是错误目标和越界行为的最强驱动因素（B0→B3，Safe Success从67.9%降至8.6%）；意图清晰度（S）影响较弱；爆炸半径（R）几乎无影响。
3. **非行动的本质**：当代理不行动时，几乎从不明确拒绝；主要是问澄清问题（ASK）或静默失败（DEFER）。问问题主要针对目标不明确（B），而非意图模糊或爆炸半径。
4. **控制面差异**：越界风险高度集中在共享控制平面（部署与流量控制、基础设施与可观测性）上，越界率高达59.8%-77.2%；而针对有界对象的操作越界率较低（≤38%）。
5. **整体数据**：55.8%-67.8%的运行至少违反一个边界；DeepSeek-v4配置行动率最高（83.0%），但也同时导致最高越界率（46.3%）和最高错误目标率（31.8%）。

## 7. 优点

- **创新的基准设计**：首次将指令模糊性作为受控变量，分离了任务难度与模糊性，使代理的猜测行为可归因。
- **细粒度的度量指标**：不仅区分完成/未完成，还区分安全成功、错误目标、越界、澄清、拒绝、推迟，比传统完成度评分更全面。
- **确定性神谕**：无需LLM作为评判者，所有边界测试完全基于可复现的副作用对比，提高了结果的可信度和可复现性。
- **基于真实事件**：每个任务族锚定于实际事故、CVE或工具行为，增强了现实相关性。
- **跨模型与支架对比**：同时评估了不同模型和不同支架的影响，揭示了模型能力与支架设计如何共同决定边界遵守行为。

## 8. 不足与局限

- **抽象差距**：实验在容器化模拟环境中进行，缺少真实生产系统中的人类审批、组织上下文、更丰富的状态和工具。因此报告的概率应视为自主路径下的下限压力测试，而非生产事故率的预测。
- **单一安全操作**：每个任务族只定义一种安全操作，代理可能采取合理替代方案但未被神谕认可；最小范围的定义在某些任务中可能有争议。
- **覆盖范围有限**：基础设施控制面仅含3个任务族，统计上不够稳健；三轴未穷尽所有模糊类型（如时间、环境、策略模糊）。
- **模型和支架快照**：仅评估了5种配置，且为2026年发布时的特定版本，随着模型更新结果可能变化。
- **未披露计算资源**：缺乏明确的硬件和算力说明，不利于其他研究者复现或扩展实验。
- **非操作分类依赖LLM**：虽然人工验证显示高度一致，但LLM judge本身可能存在偏差，尤其在少量样本上。
