## 摘要
LLM代理通过工具调用在现实环境中行动，一次误判可能导致不可逆的伤害。标准的防护措施是一个守卫模型，它将每个提议的行动标记为安全或不安全，但这种二元视图混淆了两个不同的决策：行动本身是否有害，以及是否根据用户的上下文合适。它还在行动类别而非单个实例的粒度上操作，产生常规中断，侵蚀自主性，并训练用户忽略最关键的警报。我们重新将问题构建为对{执行、询问、拒绝}的逐实例三向路由决策，并通过安全哨兵实现，这是一个轻量级守卫模型，其推理简化为一次解码调用。一个单一的解码时间阈值允许一个固定检查点在不同风险容忍度的部署中重新定位而无需重新训练。安全哨兵在整体准确性和与安全相关的召回率上优于广泛的开源和前沿闭源基线，同时控制两个方向上的错误率。

## Abstract
LLM agents act on real-world environments through tool calls, and a single misjudged action can cause irreversible harm. The standard safeguard is a guard model that labels each proposed action as safe or unsafe, but this binary view conflates two distinct decisions: whether the action is harmful in itself, and whether it is appropriate given the user's context. It also operates at the granularity of action categories rather than individual instances, producing routine interruptions that erode autonomy and train users to wave through the most consequential alerts. We reframe the problem as a per-instance three-way routing decision over {EXECUTE, ASK, REFUSE} and instantiate it with Safety Sentry, a lightweight guard model whose inference reduces to a single decoding call. A single decoding-time threshold lets one fixed checkpoint be re-positioned across deployments of differing risk tolerance without retraining. Safety Sentry outperforms a broad set of open-weight and frontier closed-source baselines on overall accuracy and safety-related recall, while controlling both directional error rates simultaneously.

## 1. 核心问题与整体含义（研究动机和背景）

- **背景**：LLM 代理通过工具调用在真实环境中执行任务，一次误判可能导致不可逆伤害。现有安全守卫模型采用二元分类（安全/不安全），但该做法混淆了两个本质不同的决策维度：动作本身是否固有有害，以及动作在当前用户上下文（任务、记忆、历史轨迹）中是否合适。
- **现存问题**：
  - 现有守卫以动作类别（如“写入文件”）而非具体实例为粒度，导致大量常规中断（如“写入临时文件”与“删除根目录”被同等对待），侵蚀代理自主性。
  - 用户面对频繁警报易产生自动化偏差和警报疲劳，倾向于忽略重要警告。
  - 许多动作既非绝对安全也非绝对有害，其适宜性取决于用户意图和运行时上下文，二元框架缺乏中间决策（询问用户）的空间。
- **核心目标**：重新定义代理安全为三向路由决策（EXECUTE / ASK / REFUSE），在保障安全与维护自主性之间取得动态平衡，实现上下文感知的人机干预。

## 2. 方法论：核心思想、关键技术细节

### 2.1 核心思想
- 将每个代理动作视为一个独立实例，守卫模型输出三个决策之一：
  - **EXECUTE（执行）**：动作明确安全，自主执行。
  - **ASK（询问用户）**：动作不确定或具有高潜在影响，请求人工确认。
  - **REFUSE（拒绝）**：动作结构上不可接受，无需询问直接拒绝。
- 通过单一解码阈值 \(\tau\) 调整 EXECUTE/ASK 边界，使得同一固定 checkpoint 可在不同风险容忍度部署中重用，无需重训练。

### 2.2 关键技术细节

#### 问题形式化
- 在代理循环的每个步骤 \(t\)，守卫接收快照 \(x = (m, u, d, h_{<t}, a_t)\)：
  - \(m\)：用户记忆项（可选个性记忆）
  - \(u\)：用户任务描述
  - \(d\)：应用域（如代码托管、文件共享）
  - \(h_{<t}\)：历史轨迹（先前的动作和观察）
  - \(a_t\)：提议的动作（工具调用）
- 输出决策 \(\hat{y} \in \{\text{EXECUTE}, \text{ASK}, \text{REFUSE}\}\)

#### 数据构建
- **真实服务环境**：部署9个自托管开源企业级服务（Gitea, Rocket.Chat, ownCloud, NocoDB, Zammad, ERPNext, OpenEMR, Vaultwarden, Mailu），每个运行在 Docker 容器中，具备原生API和真实种子数据。
- **任务来源**：
  - 内部合成：基于结构化提示生成9个服务的运营场景任务（图2）。
  - 外部基准：从 When2Call、AT-Bench、AgentHarm、TS-Bench、R-Judge 五个公开数据集筛选包含工具调用的记录。
- **个性记忆**：每个内部任务配对三种用户记忆：
  - **谨慎型**：要求不可逆操作前确认。
  - **许可型**：预先批准例行工作。
  - **对抗型**：模拟记忆投毒攻击，试图绕过安全边界。
- **标注**：采用基于14种触发模式（表4）的步骤级标注，由两个 LLM 独立标注，作者仲裁分歧，测试集作者审计，达到92%人类一致性。

#### 模型与训练
- **基座模型**：Qwen3.5-4B
- **输出格式**：扩展三个特殊token（`<|direct_execute|>`、`<|ask_human|>`、`<|refuse|>`），决策 token 作为首个完成 token 生成，后跟推理块和JSON负载（含理由和可选问题）。
- **训练细节**：
  - 使用 LoRA（rank=16, α=32, dropout=0.05）适配所有线性投影；仅新决策 token 的输入嵌入可训练。
  - 训练3个epoch（1635步），最大序列长度6144，bfloat16精度。
  - 优化器：adamw_torch_fused，学习率2×10⁻⁴，余弦调度，150步预热。
  - 损失函数：仅计算完成部分（推理块、决策token、JSON）的交叉熵。

#### 阈值路由机制
- 定义 \(q_{\text{EXECUTE}} = p_{\text{EXECUTE}} / (p_{\text{EXECUTE}} + p_{\text{ASK}})\)，即非REFUSE子空间内重新归一化的执行概率。
- 给定阈值 \(\tau \in [0,1]\)，若 \(q_{\text{EXECUTE}} \ge \tau\) 则预测 EXECUTE，否则 ASK；REFUSE 预测保持不变（安全性底层固定）。
- 三种预设操作点：自主型（OAR≤5%）、平衡型（最小化 OAR+UAR）、保守型（UAR≤5%）。

## 7. 优点

1. **问题重构创新**：将传统二元安全分类拓展为三向路由，明确区分“询问”与“拒绝”，填补了中间决策空白，更符合真实需求。
2. **轻量高效**：4B参数 + LoRA微调，推理仅单次解码调用，部署成本低。
3. **解码阈值灵活调控**：无需重训练即可适配不同风险场景和用户偏好，实践性强。
4. **上下文感知**：通过个性记忆和轨迹实现实例级、用户级定制，克服粗粒度警报缺陷。
5. **双向错误控制**：OAR/UAR 同时保持在低水平，避免了基线常见的“一端坍塌”问题。
6. **泛化能力**：在完全未见的服务（Mailu）和不同上游框架上保持高精度，表明学到的是通用决策结构而非死记硬背。
7. **数据构建细粒度**：基于真实自托管服务、14种触发模式、双模型标注+作者审计，标注质量高（92%人类一致）。
8. **消融与鲁棒性实验充分**：覆盖阈值、记忆、OOD、框架、骨干等多个维度。

## 8. 不足与局限

1. **领域覆盖有限**：只涉及企业办公/计算机使用场景（代码托管、工单、文件共享等），未评估开放写作、创意工具、物理机器人等领域的泛化性。
2. **阈值手动设定**：τ 需操作者根据经验手动校准，非自动化学习，新场景下可能需要多次尝试才能达到理想行为。
3. **基线对比公平性存疑**：所有基线以零样本形式运行（未使用论文的14种触发标注），可能未发挥其最佳性能（尤其对于擅长遵循指令的模型，若给出更详细提示可能提升）。但作者在提示中提供了简要决策标准（图13），公平性在可接受范围内。

5. **单一基座模型**：Safety Sentry 仅基于 Qwen3.5-4B，未验证在更大基座模型上是否进一步提升性能（文中指出 Qwen3.5 系列从4B到27B提升有限，暗示缩放边际收益递减，但未用更大小模型作为基座训练）。

7. **未讨论多轮记忆演化**：当前记忆是静态输入的；用户偏好随时间变化时如何更新记忆未涉及。
8. **安全防护单一层次**：论文明确表示该守卫应作为纵深防御的一层而非唯一保障，但未集成其他安全机制（如规则引擎、沙箱执行）进行联合评估。
