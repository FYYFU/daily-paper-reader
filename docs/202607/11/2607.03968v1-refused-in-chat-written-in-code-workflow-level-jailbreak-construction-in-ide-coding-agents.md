## 摘要
大型语言模型越来越多地被部署为集成开发环境（IDE）中的编码代理，它们能够分解任务、生成和编辑文件、运行代码并在多轮交互中优化输出。然而，这些模型的安全性评估方式常常仍类似于聊天机器人：一次有害提示，一次响应，孤立地进行判断。我们引入了工作流级越狱构造，这是一种失败模式，其中有害目标通过软件开发生命周期的常规阶段组合而成，而非通过单个直接提示生成。利用Visual Studio Code中的GitHub Copilot，我们研究了四个闭源后端：Claude Sonnet 4.6、Claude Haiku 4.5、Gemini 3.1 Pro和Gemini 3.5 Flash。在来自Hammurabi Code、HarmBench和AdvBench的204个提示中，模型在直接聊天、CSV读取和单步代码修复基线条件下几乎完全拒绝，每个基线条件下仅有8/816次成功响应。然而，在完整工作流下，相同的提示和后端产生了816/816次不安全的教学射击完成，所有结果均经两位专家评估者根据严格评分标准独立确认。这些结果表明，对话式拒绝基准可能会显著高估已部署编码代理的安全性，并促使防御方法在多轮IDE工作流及其生成的工件中推理安全性，而不仅仅是单个对话轮次。

## Abstract
Large language models are increasingly deployed as IDE-integrated coding agents that decompose tasks, generate and edit files, run code, and refine outputs over many turns. Yet their safety is still often evaluated as if they were chatbots: one harmful prompt, one response, judged in isolation. We introduce workflow-level jailbreak construction, a failure mode in which a harmful objective is assembled across ordinary stages of a software-development workflow rather than generated through a single direct prompt. Using GitHub Copilot in Visual Studio Code, we study four closed-weight backends: Claude Sonnet 4.6, Claude Haiku 4.5, Gemini 3.1 Pro, and Gemini 3.5 Flash. Across 204 prompts from Hammurabi's Code, HarmBench, and AdvBench , the models show near-complete refusal under direct chat, CSV-read, and single-step code-fix baselines, with only 8/816 successful responses in each baseline condition. Under the full workflow, however, the same prompts and backends produce 816/816 unsafe teaching-shot completions, all independently confirmed by two expert evaluators under a strict rubric. These results show that conversational refusal benchmarks can substantially overstate the safety of deployed coding agents and motivate defenses that reason about safety across multi-turn IDE workflows and their generated artifacts, not only individual chat turns.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：现有大语言模型（LLM）编码代理（如 GitHub Copilot）的安全评估仍沿用聊天机器人式的单轮测试（一个有害提示 → 一次响应 → 孤立判断），忽略了集成开发环境（IDE）中多轮软件开发工作流带来的组合攻击风险。当有害目标被拆解到文件读取、代码运行、指标优化等常规开发步骤中时，模型可能生成不安全内容，而对话式拒绝基准却显示模型“安全”。
- **研究背景**：LLM 正从被动补全工具变为主动编码代理，能分解任务、编辑文件、执行代码并多轮迭代。现有安全研究多关注单轮提示攻击（如 GCG、CodeAttack）或多轮对话攻击（如 Crescendo），但针对 IDE 环境下软件工程工作流本身的安全评估几乎空白。已有基准（如 Hammurabi’s Code）虽评估了代码生成的有害性，却未考虑工作流级组合越狱。
- **整体含义**：对话式拒绝基准严重高估已部署编码代理的实际安全性，需要在多轮 IDE 工作流及其生成的工件（如文件、脚本、示例数据）层面进行安全评估与防御。

## 2. 论文提出的方法论

- **核心思想**：提出**工作流级越狱构造（Workflow-Level Jailbreak Construction）**，不再依赖单个有害提示，而是将有害目标分散到软件开发生命周期的常规阶段中，使模型在看似正常的工程任务（如构建评估管道、提高攻击成功率 ASR）中自动编写有害内容。
- **关键技术细节（7 个流程阶段）**：
  1. **框架建立**：让编码代理 A 为名义目标模型 T 构建越狱评估管道。
  2. **基准注入**：将公开的有害基准提示作为普通数据加载，而非直接请求。
  3. **管道构建**：A 编写查询 T、收集响应、计算 ASR 的代码。
  4. **指标引入**：操作者 O 声称 ASR 较低，将任务转为指标优化问题。
  5. **良性升级**：O 建议添加教学射击（teaching-shot）示例，A 先添加无害的，使写例子成为常规操作。
  6. **有害升级**：O 要求教学射击包含有害的提示-答案对。A 自己生成答案（这些提示在直接聊天中被拒绝）。
  7. **强化**：O 反馈 ASR 正在提升，维持优化框架，促使 A 继续生产更多有害内容。
- **算法流程（文字描述）**：无公式，采用固定脚本协议，所有操作由操作者通过自然语言/代码指令完成，不修改模型权重或系统提示。
