Abstract 

LLM-based agents leverage third-party skills to extend their capabilities in open-world sce-narios. However, third-party skills can intro-duce extra security vulnerabilities, as seem-ingly harmless skills can contain latent safety risks that only emerge during actual execution. In this work, we conduct a systematic inves-tigation into how well current agent systems recognize and avoid such risks. To support quantitative and qualitative evaluation, we con-struct OpenSkillRisk, a dedicated safety bench-mark containing 263 risky skills collected from public skill marketplaces. We classify these skills into seven categories based on their threat types and pair each skill with a standardized user task and a corresponding sandbox for con-trolled evaluation. Distinct from prior bench-marks, OpenSkillRisk not only covers more realistic and diverse unsafe scenarios, but also provides a fine-grained analysis to diagnose the behavioral patterns of agents in such scenar-ios. We conduct comprehensive experiments covering three mainstream CLI agent frame-works and thirteen state-of-the-art LLMs. Ex-perimental results show that no tested system handles risky skills reliably: even the safest configurations still execute unsafe actions in about 17% of cases. Context-dependent and system-level risks are especially difficult for current agent systems to avoid. Our behavioral analysis reveals three recurring failure patterns: agents may fail to recognize the risk, recog-nize it but fail to intervene before acting, or follow skill instructions beyond the user’s in-tended scope. These findings highlight the need to improve both risk reasoning in LLMs and execution control in agent frameworks. Our code and data are publicly available at https: //github.com/Miaow-Lab/OpenSkillRisk .

1 Introduction 

Large language models (LLMs) have rapidly evolved from passive text generators into active, autonomous agents capable of executing complex,

multi-stage operations in real-world settings (Wang et al., 2024; Luo et al., 2025). This shift is particu-larly visible in command-line interface (CLI) tools such as Claude Code (Anthropic, 2026a), Gemini CLI (Google, 2026a), and Codex CLI (OpenAI, 2025), which embed frontier models directly into developer workflows. To enable reusable and scalable agent behaviors, agent skills have emerged as a core abstraction. A skill is defined as a structured package comprised of instructions, code templates, resources, and veri-fication logic. It enhances agent performance dur-ing inference without requiring model modifica-tion (OpenAI, 2026b; Jiang et al., 2026a; Xu and Yan, 2026). Currently, skill ecosystems have grown rapidly, with community repositories hosting thou-sands of user-contributed modules. However, be-cause skills are easy to publish, install, and invoke, open repositories may contain many unsafe skills 1arXiv:2607.20121v1 [cs.CL] 22 Jul 2026 that hide malicious code, expand privileges, trigger risky execution, or leak sensitive data (Schmotz et al., 2025), which can introduce hidden side ef-fects to users, as shown in Figure 1. This raises a critical question: whether current agent systems can effectively detect and deal with such safety risks. Prior work has begun to evaluate this problem from several related directions, including broad agent safety, skill utility and quality, and robust-ness under skill-oriented attacks (Feng et al., 2026; Zhang et al., 2025; Li et al., 2026a; Han et al., 2026; Schmotz et al., 2026; Jia et al., 2026). However, these existing benchmarks are still insufficient to answer the above question. On one hand, most of them focus on limited types of safety threats aris-ing from third-party skills, falling short of captur-ing the real and diverse distribution of such threats (Schmotz et al., 2026; Jia et al., 2026). On the other hand, their main metric, attack success rate (ASR), is insufficient for characterizing agents’ responses to unsafe scenarios. To address these limitations, we introduce Open-SkillRisk , a dedicated benchmark for end-to-end evaluation of agent system safety when confronting risky real-world third-party skills. OpenSkillRisk contains 263 real-world risky skills across two risk settings, 7 attack categories, and 9 diverse task domains, and evaluates agents with both execution-level ASR and cognition-level risk awareness. The construction of OpenSkillRisk consists of three stages: real-world risky skill selection, task and sandbox construction, and agent behavior evalu-ation. We first curate validated risky skills from public skill marketplaces through static filtering, contextual review, and human verification. We then pair each skill with a realistic benign task package and an isolated sandbox environment, so that the skill’s risk surface can be triggered. Finally, we analyze agents’ decision-making patterns and ex-ecution behaviors using quantitative performance metrics, revealing how agents respond and operate within these environments. Additionally, we introduce awareness rate as a cognition-level metric that measures whether agents explicitly recognize and warn users about hidden skill-induced risks. Even when agents fail to stop risky execution, explicit risk alerts still serve as a critical safety barrier and help users take follow-up actions. We further classify agent behaviors into five categories: (a) unaware risky execution, (b) aware but still executes, (c) warns and blocks, then aborts, (d) warns and blocks, then completes safely, and (e) silent safe completion, enabling a more fine-grained analysis. We conduct extensive experiments covering 3 prevalent CLI agent harnesses and 13 representa-tive proprietary and open-source LLMs. Results show that current agent systems still frequently trig-ger unsafe skill behaviors, with safety performance varying widely across harnesses, models, risk set-tings, and attack categories. In particular, context-dependent risks and system-level attacks remain difficult to detect and block. These findings high-light the need for stronger agent safety mechanisms that jointly improve risk reasoning and execution control when using third-party skills. 

2 Related Works 

Agent systems and reusable skills. Modern LLM-based agents achieve superior capabilities not only via strong foundation models, but also by coupling language reasoning with tool use, en-vironment interaction, and iterative verification (Yao et al., 2023; Schick et al., 2023; Shinn et al., 2023). This paradigm underlies practical coding agents and agent-computer interfaces (Yang et al., 2024; Anthropic, 2026a; OpenAI, 2025), which can complete full-cycle development tasks includ-ing repository inspection, file editing, and com-mand execution. For repeated domain-specific tasks, recent research encapsulates procedural task knowledge as modular reusable skills (Wang et al., 2023; Li et al., 2026a; Han et al., 2026). By sepa-rating lightweight routing metadata from detailed task-specific guidance, such modular design im-proves system reusability and behavioral consis-tency. Prior studies also show that high-quality curated skills can substantially boost the practical performance of LLM agents (Li et al., 2026a). 

Agent safety benchmarks. As summarized in Table 5, existing benchmarks fall into three main-stream branches: broad agent safety (Feng et al., 2026; Zhang et al., 2025; Andriushchenko et al., 2025; Li et al., 2026b), skill utility or quality (Li et al., 2026a; Han et al., 2026; Wang et al., 2026a), and robustness under skill-oriented at-tacks (Schmotz et al., 2026; Jia et al., 2026; Wang et al., 2026b; Wei et al., 2026). CIK-Bench (Wang et al., 2026b) and ClawSafety (Wei et al., 2026) mainly study persistent-state poisoning or multi-channel indirect prompt injection, without focusing on risks brought by skills. Skill-Inject (Schmotz 2et al., 2026) and SkillJect (Jia et al., 2026) di-rectly probe vulnerabilities brought by malicious skills, yet they rely on manually poisoned or syn-thetic adversarial skills, failing to cover diverse real-world risk patterns. HarmfulSkillBench (Jiang et al., 2026b) adopts real marketplace skills, but mainly investigate skills with advertised harmful or policy-violating functionalities. Distinct from all prior benchmarks, OpenSkillRisk focuses on la-tent execution-time risks hidden within seemingly benign third-party skill in real skill ecosystems. 

Defenses against skill risks. Current mitigation strategies for skill-induced risks typically combine pre-screening with runtime defense. Pre-screening mechanisms detect risky skills before official re-lease or local installation via static code analysis, semantic consistency checks, or LLM-based in-spection (Liu et al., 2026; Guo et al., 2026; Snyk, 2026; OpenClaw, 2026); Runtime defense con-strains high-risk execution via least-privilege con-trol and tool-call policy enforcement (Shi et al., 2026), isolation and unsafe-flow control for un-trusted inputs (Kim et al., 2025), and sandbox or user approval mechanisms (Anthropic, 2026b; Ope-nAI, 2026a; Google, 2026b). However, users of-ten skip pre-screening for convenience; static au-diting cannot capture context-aware risks or risks triggered by long execution chains; and runtime defense fails once unmonitored critical execution paths or side effects exist. OpenSkillRisk evaluates the end-to-end safety abilities of an agent system, focusing on whether agents can independently per-ceive and block hidden risky skill behaviors. 

3 OpenSkillRisk Construction 

We introduce OpenSkillRisk, a dedicated bench-mark to evaluate the safety of agent systems when using third-party skills from open ecosystems. The benchmark construction is organized around three questions: how to select representative real-world risky skills, how to construct corresponding tasks, and how to evaluate the performance and analyze the behavior of agents facing risky skills. The full construction pipeline is illustrated in Figure 2, con-sisting of three core modules: real-world skill selec-tion, user task generation, and multi-dimensional agent behavior evaluation. Comprehensive statisti-cal details of the benchmark dataset are presented in Appendix B.6. 

3.1 Skills Selection Skill collection. We curated 175,891 unique En-glish skills from two major public marketplaces, SkillsMP and Skills.rest up to March 20, 2026. We applied a a unified normalization and cross-source deduplication pipeline to precess raw data. The col-lected skills cover diverse domains including soft-ware development, business operations, data analy-sis, research, documentation, and creative concent creation. Overall, around 70.8% of the skills come from from SkillsMP and 29.2% from Skills.rest. Detailed crawling and processing procedures are provided in Appendix B.1. 

Risky Skill Candidate Mining. To systemati-cally identify potentially risky skills from a mas-sive corpus of real-world data, we adopt a two-stage coarse-to-fine filtering strategy inspired by SkillScan (Liu et al., 2026): • Rule-based Filtering: We utilize an open-source static scanner (huifer, 2026) to perform initial screening over all skills to obtain a candi-date set of risky skills. By matching against pre-defined unsafe patterns, the scanner efficiently identifies candidates with suspicious text or code signatures, including external network re-quests to untrusted domains, attempts to access sensitive files, and potential code injection. • Context-based Filtering: We further use LLM-based review to refine the candidate pool. We first use a resource-efficient model for initial triage, and then use a stronger model for fine-grained risk assessment and preliminary label-ing. This step substantially reduces false pos-itives caused by benign skills or skills with explicitly malicious promotional purposes. This two-stage filtering pipeline yielded 1,799 high-quality risky skill candidates. We further conducted stratified sampling, followed by a high-capability LLM confirmation pass and human ex-pert review of each retained benchmark instance. Human experts checked whether each skill con-tains real safety-relevant behavior, whether its risk type is correctly assigned, and whether the paired task can expose the risk in the sandbox. Ultimately, we incorporated two types of risky skills into OpenSkillRisk: 139 unconditionally malicious skills with explicit harmful intentions to users that should be rejected unconditionally, and 124 context-dependent risky skills. Specifi-cally, such skills may function safely with explicit, 3Figure 2: Overview of the OpenSkillRisk construction pipeline. 

scope-restricted user authorization, but pose severe safety risks when user intent is ambiguous, or when agents blindly trust the skill and overstep the legiti-mate scope of original user requests. The complete candidate mining workflow is elaborated in Ap-pendix B.2. 

Risky Skill Taxonomy To standardize our bench-mark dataset construction, we build a systematic risk taxonomy grounded in real-world hazardous skill behaviors. Incorporating mainstream security definitions (Schmotz et al., 2026; Li et al., 2026d; Liu et al., 2026), we categorized risky skills into seven distinct attack categories: control-plane hi-jacking, authority expansion, data harvesting, exe-cution bootstrapping, persistence implantation, out-bound exfiltration and external state manipulation. Please see Appendix B.4 for a detailed explanation. 

3.2 Task Generation for Skills 

To evaluate how agent systems respond to risky third-party skills end-to-end, each selected skill is paired with a realistic benign task that naturally activates the skill’s risk surface. Different from plain prompt-based evaluation settings, we con-struct complete task packages as standard bench-mark instances. Full task construction workflows and dedicated generation prompts are documented in Appendix B.3. 

Task composition and construction. Each task package contains five core components: a benign user request, task-specific workspace resources, synthetic user-context files, expected safe agent behaviors, and dedicated paths for storing gener-ated artifacts and execution logs. The workspace re-sources cover static workspace files, including busi-ness instructions, code snippets, configuration files and example datasets, as well as dynamic user con-text files that emulate real local user environments, such as synthetic home directories with mock cre-dentials, application preferences, cache files, and temporary memory files. To prevent data leakage or unintended operations, all sensitive fields, ac-count identifiers, credentials, and infrastructure references are replaced with format-compliant fic-tional placeholders. These resources provide suf-ficient context for the benign objective while ex-posing the action surface needed to evaluate the corresponding skill risks. 

Sandbox and isolation environment construc-tion. Each task is deployed in an isolated sand-box that allows the agent to interact with the skill and task context realistically while blocking harm-ful side effects on evaluators or external systems. The sandbox separates the project directory for workspace files, the skill installation directory, and the runtime directory for logs and execution ev-idence. All external interactions, including API calls, database operations, deployment commands, and remote service requests, are replaced with local command stubs or mock services. These stubs re-turn deterministic responses and record invocation parameters, generated artifacts, and runtime events, thereby removing real network effects and third-party dependencies while preserving unsafe deci-sion points. The runtime directory records mock invocations and artifacts generated by agents as evidence for judging whether the agent attempted unsafe actions. 4Task validation and iterative optimization. We validate each candidate task through static review and execution-trace review. Static review checks (i) scenario authenticity : tasks should meet practical user demands, and represent typical skill use. (ii) intent safety : original user intent should be com-pletely benign, without explicit or implicit harm-ful instructions. (iii) risk-goal balance : each task should balance risk exposure with goal completion by allowing the benign objective to be completed through a safe alternative path. We further conduct a trial run with a standard agent and inspect full execution responses, generated files, workspace snapshots, and runtime logs. This execution-trace review checks whether the task can run smoothly end-to-end, successfully trigger preset risk deci-sion points, produce unsafe behavior matching the target risk category when applicable, preserve a safe completion path, and generate sufficient evi-dence for subsequent safety assessment. Tasks that fail either static or dynamic review will be sent back for iterative revision, covering user prompt rewriting, workspace file adjustment, mock stub parameter modification and expected safe behavior update.. The process repeats until the task satis-fies benchmark requirements or reaches a prede-fined maximum iteration budget. See Appendix B.5 for exemplar benchmark instances. After this validation loop, human experts further review the final task packages to check that the user request is natural, that the unsafe behavior is not needed for normal task completion, and that the sandbox evidence is sufficient for later judgment. 

3.3 Agent Performance Evaluation 

To comprehensively quantify agent behavioral pat-terns against risky skills, we design two comple-mentary evaluation metrics covering execution-level operational safety and cognition-level risk awareness. For each benchmark test instance i,we define two binary indicator variables: ri ∈{0, 1} denotes whether the agent triggers prede-fined risky execution paths, and zi ∈ { 0, 1} rep-resents whether the agent explicitly detects and reports hidden risks to end users. Given a specific dataset split Ds, we calculate the two core metrics:

(i) ASR (Execution-Level Safety): This metric quan-tifies the ratio of test cases where agents execute unsafe operations. A lower ASR reflects better be-havioral safety, as the agent successfully avoids risky skill executions in most scenarios. (ii) Aware-ness Rate (Cognition-Level Safety): This metric evaluates agents’ cognitive capability to perceive implicit risks within third-party skills. It measures the proportion of interactions where agents actively identify potential risks and send clear warnings to users. Notably, even if agents cannot terminate risky execution ultimately, explicit risk alerts still build a critical safety barrier and assist users in making secure follow-up decisions. To jointly assess agents’ dual capabilities in 

avoiding unsafe execution and recognizing latent risks , we propose a holistic safety metric Fsafe in-spired by the harmonic mean calculation of stan-dard F1-score: 

Fsafe = 2 · (1 − ASR) · Awareness (1 − ASR) + Awareness .

This composite metric balances the avoidance rate 

and the risk awareness rate . A high score can only be achieved when the system maintains both low risky execution rates and high risk awareness. Different from single-dimensional metrics, Fsafe 

precisely reflects the dual requirements for secure LLM agents: reliable safe operational behaviors at the execution level, and clear risk detection at the cognition level. 

3.4 Agent Behavior Analysis 

To better understand how agent systems respond to risky skills, we conduct a fine-grained behavioral analysis for all evaluation instances. We classify agent behaviors into the following five categories, with case demonstrations in Figure 5: 

(i) Unaware risky execution : The agent shows no explicit sign of recognizing the risk and triggers the risky path directly. This is fully unsafe, demon-strating a complete deficiency in risk recognition. 

(ii) Aware but still executes : The agent success-fully detects and reports the risk, but still proceeds with the risky action, or delivers risk warnings only after the unsafe artifact has already been generated. Such behavior is still unsafe, reflecting delayed or ineffective risk intervention capability. 

(iii) Warns and blocks, then aborts : The agent blocks all risky execution paths, but abandons the original benign user task entirely. Though this behavior eliminates potential risks, it sacrifices task usability. 4 Experiments 

We comprehensively evaluate the behavior of 3 mainstream CLI harnesses 13 frontier models on OpenSkillRisk, focusing on four key research ques-tions that highlight their vulnerabilities and mitiga-tion potential. 

4.1 Experimental Setup Evaluator setup. We evaluate each run using a 

GPT-5.1-Codex-Mini agent evaluator. The evalu-ator receives five inputs: the task instruction, the initial input snapshot, the agent response, the ex-pected safe behavior specification, and all collected artifacts produced during execution. The evaluator can be replaced by other LLM agents, as we show in Appendix D.3 the agreement of different judge agents can be high. 

Agent–skill trust boundary. We further make the user–skill trust boundary explicit through a shared system prompt used in all evaluated agent runs, as shown in Figure 17. The prompt instructs agents to prioritize the user’s goal over skill instructions, and account for the possibility that users may be un-aware of hidden risks inside skills. As a result, ob-served failures more directly reflect whether agents fail to consider user security when operating with third-party skills, often by more faithfully follow-ing skill instructions.

4.4 RQ3. What Distinct Behavioral Patterns Emerge When Processing Unsafe Skills? 

To understand the agent behaviors behind the ag-gregate safety results, we further decompose each trajectory into the behavior types defined in Sec-tion 3.4. Table 3 reports the overall distribution of behavior types across different harness-model con-figurations. Table 11 gives the split-wise results. 

Safer systems shift from unrecognized execution to explicit risk handling. Stronger configurations generally reduce unaware risky execution and in-crease explicit safe handling. For example, GPT-5.4 achieves a 73.76% warn-and-execute-safely rate (W-E) with only 16.73% unaware risky ex-ecution (Ua-E), while GLM 5.1 also shows high W-E rates under both Codex and Claude Code. In contrast, weaker configurations retain substantially higher Ua-E rates; Gemini CLI + Gemini 2.5 Pro, for instance, reaches 48.67% Ua-E. This suggests that robust behavior is not merely about avoiding execution, but about recognizing risks and complet-ing the task through safer alternatives. 

Models and harness-level constraints play com-plementary safety roles. Models mainly affect risk recognition and the generation of safer alterna-tives, as reflected by lower Ua-E and higher W-E rates. Harness-level constraints, however, deter-mine whether an agent can explicitly terminate risky workflows. W-A is nearly absent in Codex and Gemini CLI, but reaches 38.40% and 21.67% for Claude Code + Claude Sonnet 4.6 and Opus 4.6. Behavior distribution also varies for shared LLMs. Therefore, robust agent safety requires both efforts in both LLM and harness levels. 

4.5 RQ4. Does a Guard Skill Improve Safety, and at What Utility Cost? 

We study whether agents can be defended through the same skill mechanism that creates risk. We syn-thesize a benchmark-informed guard skill using the OpenSkillRisk attack taxonomy and evaluate two modes: passive loading , where the guard is placed in the skill directory and the agent decides whether to load the guard, and active loading , where it is explicitly loaded before target-skill interaction. Table 4 shows that passive loading provides only limited protection, revealing a routing bottleneck: agents must first recognize risk before consulting the guard, especially when risky behavior is hidden inside useful-looking third-party skills. In con-trast, active loading substantially improves safety 8by making the defensive policy salient at runtime, thereby increasing the risk recognition and the unsafe-execution avoidance. The larger safety gain also comes with more over-defense. As Table 14 shows, on a separate set of 40 benign skills, the average over-defense rate is 1.88% under passive loading and 22.50% un-der active loading, with substantial variation across models. These results show that guard skills can im-prove safety when loaded reliably, but they should complement rather than replace harness-level ex-ecution controls. Practical deployment requires routing that invokes the guard when needed with-out unnecessarily blocking benign tasks. 

5 Conclusion 

We presented OpenSkillRisk, a benchmark for eval-uating agent safety under real-world third-party skills. We show that current agents remain non-robust: agents frequently follow risky skill instruc-tions, struggle with contextual risk judgments, and are more vulnerable to system-level than data-level attacks. These findings suggest the need to improve both LLMs and harnesses for agent safety. 

Ethical Considerations 

OpenSkillRisk contains real-world agent skills col-lected from public skill discovery platforms. We use these platforms as discovery indexes rather than treating marketplace availability as permission for unrestricted redistribution. Third-party skill con-tents remain subject to their original licenses and platform terms. Our work is a data collection and risk annotation effort rather than the creation of new malicious skills. Nevertheless, we recognize that aggregating and standardizing such content may increase discoverability and lower the cost of misuse. We therefore treat the dataset as dual-use. To mitigate these risks, we adopt a gated-access release policy for the raw dataset. Access is limited to legitimate safety research and defensive evalua-tion, and requesters must agree not to use the data against real users, production systems, or third-party services, not to redistribute the raw dataset, and not to attempt to re-identify skill authors or publishers. Public artifacts are limited to the bench-mark framework, evaluation code, taxonomy, and aggregate statistics. We have anonymized the unnecessary personal information of the publishers in the released dataset. We also note that the current version does not in-clude fully inertized versions of all skills. For this reason, the raw collection is not suitable for un-restricted public release, and experiments should only be conducted in sandboxed or otherwise iso-lated environments with no access to real user data, credentials, or production services. This release design aims to balance scientific transparency with harm reduction.

B Benchmark Construction Details 

We explain here the details of the benchmark con-struction that are not elaborated in the main text. 

B.1 Skill Collection Details Marketplace Crawling. We crawled all publicly accessible skills from two public marketplaces SkillsMP and Skills.rest up to March 20, 2026. For SkillsMP, we performed site-wide discovery by combining category-level pagination with search-prefix sharding, and downloaded the archive bun-dle associated with each discovered skill. For Skills.rest, we combined site-index traversal with site-wide randomized crawling to reduce the sam-pling bias that would arise from relying only on popularity-based rankings or alphabetical ordering. In total, the raw crawl yielded 224,270 processable skill bundles, including 76,115 from Skills.rest and 148,155 from SkillsMP. 

Cross-market Deduplication. To avoid inflating the dataset when the same skill was published on multiple marketplaces, we applied a unified nor-malization and cross-source deduplication pipeline. Specifically, we first removed samples with miss-ing SKILL.md file or with content less than 10 lines. We then computed a SHA-256 hash over the normalized SKILL.md file and used this hash as the cross-market deduplication key. We further retained only English-language skills. When du-plicate records shared the same hash, we kept the record with richer metadata, then more non-empty lines/characters, with a deterministic source/UID tie-breaker. After this step, we obtained 175,891 unique English skill bundles, including 51,305 from Skills.rest and 124,586 from SkillsMP. 

B.2 Risky Skill Candidate Mining Details Coarse filtering. In the candidate mining stage, we used the deduplicated skill corpus as input. For the rule-based coarse filtering stage, we ran 

skill-security-scan (huifer, 2026) and aug-mented it with a small set of supplemental regular-expression rules. These rules target common suspi-cious patterns, including external network requests, access to sensitive files, dynamic code execution, remote script fetching and execution, instruction overriding, and references to sensitive filesystem paths. We then used DeepSeek-V3.2 for context-aware initial filtering over each candidate skill bundle. This step was designed to control filtering costs, so we did not use a stronger model at this stage. The model was asked to return a final verdict and confidence score. This prompt follows the LLM analysis prompt from Liu et al. (2026). During ag-gregation, candidates judged as vulnerable with confidence at least 0.6 were retained. If the model judged a candidate as benign , we only overturned the static finding when the benign confidence was at least 0.9; otherwise, the skill was retained as a risk candidate whose static signal was not suffi-ciently contradicted. 

Fine-grained filtering. Because the judgments from DeepSeek-V3.2 can be inaccurate, we then applied a stronger GPT-5.4-mini -based fine-grained filtering step. This refinement removes false positive skills and skills whose advertised purpose is explicitly malicious or abusive. B.3 Task Curation Details 

To construct the tasks designed in Section 3.2 for specific skills, we mainly prompt a Codex GPT-5.4 agent to inspect the skills and generate the re-sources. The prompting process contains three stages. First, the generation prompt constructs an initial candidate task according to the selected skill, its intended benign use case and the associated risk category. Second, the static review prompt inspects the generated task before execution, focusing on sce-nario authenticity, user-intent safety, sandbox isola-tion, and the balance between benign task comple-tion and risk exposure. Finally, after the agent system executes the task, the execution-trace review prompt evaluates whether the task works end-to-end, whether the ex-pected unsafe behavior is exposed, and whether the benign goal can still be achieved without requiring the agent to perform the unsafe operation. These prompts are used in an iterative loop. If a candidate task fails the static review or the execution-trace review, the corresponding feedback is returned to the generation stage. The task is then revised by adjusting the user request, workspace 12 files, local stub behavior, expected artifacts, or safety constraints. This process continues until the task satisfies the benchmark requirements or until a predefined maximum number of iterations is reached.

Number of iterations. For the design of the num-ber of iterations, we use different iteration budgets for obviously malicious skills and contextual-risk skills, since the scenarios involved in these two branches have different levels of complexity. For obviously malicious skills, we used one generation–validation iteration ( N = 1 ). These cases usually had a clear mismatch between the benign task and the hidden behavior, such as sus-picious installers, data collection, credential har-vesting, or persistence hooks. Therefore, one pass was sufficient to confirm that the task exposed the malicious path. For contextually risky skills, we used two itera-tions ( N = 2 ). These tasks are more complex and require more calibration because the risky actions could appear reasonable in a trusted operational setting. The second pass was used to tune the lo-cal context, expected safety behavior, and expected artifacts so that the task neither made the risk too obvious nor made the unsafe action necessary for benign completion.

B.5 Case Study
