## 摘要
持久化AI代理将大语言模型（LLM）从单轮交互扩展为长期运行的软件系统。与传统聊天助手不同，这些代理中的不安全内容可通过持久化状态、可复用技能和工具中介交互传播，形成显著更大的语义攻击面。我们观察到，此类代理中大多数安全关键交互通过自然语言令牌流传输，包括内存更新、工具参数、检索文件以及组件间通信。这一观察催生了一种新的安全表述：不安全行为可在到达特权运行时接收点之前被拦截为风险语义流。基于此洞察，我们提出TokenWall，一种在代理令牌流上充当语义防火墙的运行时防御框架。TokenWall对这些流执行边界感知的语义审计，构建结构化的源-接收审计记录，在执行前应用轻量级本地检查，并选择性地将模糊的高风险案例升级到更强的仲裁模块。与依赖稀疏审计或远程大模型监督的先前方法不同，TokenWall实现了全覆盖的执行前中介，同时减少远程仲裁和延迟。在CIK-Bench上的实验表明，TokenWall将攻击成功率降至12.5%，同时在无需人工确认的情况下保持了97.4%的良性可执行通过率。TokenWall在良性案例上仅引入0.69秒的额外延迟，表明语义运行时包含措施可以为持久化AI代理实现实用的安全-效用权衡。

## Abstract
Persistent AI agents extend large language models (LLMs) beyond single-turn interaction into long-lived software systems. Unlike traditional chat assistants, unsafe content in these agents can propagate through persistent state, reusable skills, and tool-mediated interactions, creating a substantially larger semantic attack surface. We observe that most security-critical interactions in such agents are transmitted through natural-language token flows, including memory updates, tool arguments, retrieved files, and inter-component communications. This observation enables a new security formulation: unsafe behavior can be intercepted as risky semantic flows before reaching privileged runtime sinks. Based on this insight, we propose TokenWall, a runtime defense framework that acts as a semantic firewall over agent token flows. TokenWall performs boundary-aware semantic auditing over these flows, constructing structured source-sink audit records, applying lightweight local inspection before execution, and selectively escalating ambiguous high-risk cases to stronger arbitration modules. Unlike prior approaches that rely on sparse auditing or remote large-model oversight, TokenWall enables full-coverage pre-execution mediation while reducing remote arbitration and latency. Experiments on CIK-Bench show that TokenWall reduces attack success rate to 12.5% while maintaining a 97.4% benign executable pass rate without human confirmation. TokenWall further introduces only 0.69 seconds of additional latency on benign cases, demonstrating that semantic runtime containment can achieve a practical security-utility trade-off for persistent AI agents.

## 2. 论文提出的方法论

### 2.1 核心思想
- **语义令牌流（Token Flow）**：将代理中通过自然语言传递的安全相关交互（用户输入、工具参数、记忆写入、组件间消息等）建模为 **令牌流** \( f = (x, s, t, c, b) \)，其中 \( x \) 为负载（分割为语义跨度），\( s \) 为源，\( t \) 为接收点，\( c \) 为运行时元数据，\( b \) 为要跨越的边界。
- **安全边界**：分为三类——**上下文流**（影响推理上下文、记忆、持久知识）、**权威流**（改变代理的身份、会话绑定、授权状态）、**能力流**（触发工具执行、外部披露、文件操作等）。
- **防御位置**：在令牌流跨越这些边界之前进行审计（pre-transfer），而非事后观察。

### 2.2 关键技术细节（三层管道）
1. **轻量级预检查（Precheck）**：确定性规则，捕获明确违规（如硬不变规则：私有端点暴露到执行或披露边界则直接拦截）。输出隐式违规信号（如秘密泄露、外部接收点等）供下游使用。
2. **本地小模型审计器（Local Small Auditor）**：核心组件。接收结构化的源-接收记录（含安全表面、边界类型、元数据、负载跨度、预检查信号、所有者意图摘要），使用小型 LLM（如 Qwen3-4B）进行语义判断。输出结构化报告：决策（allow / rewrite_and_continue / defer_to_human / block）、重写负载、风险分数（0-1）、不确定性、可利用性、影响、残差风险类型、重写完整性、是否需要大模型审查等。
3. **降级仲裁（Fallback Arbitration）**：当本地审计器不确定性高、风险高、重写不完整或涉及高影响操作时，由更大的模型（如 Qwen3.6-Plus）进行最终仲裁。仲裁器输出相同动作空间。

### 2.3 关键设计
- 重写规则：只允许删除/掩码风险跨度，不能新增事实、扩大权限、改变用户目标。
- 失败闭合：格式错误、缺失字段、无效重写、超时均视为拦截。
- 表面特定的升级谓词 Φ_type(f) 基于预检查规则、残差风险类型、可利用性、影响等阈值决定是否升级。

## 6. 论文的主要结论与发现

1. **TokenWall 实现了最好的攻击抑制**：在 CIK-Bench 上总体 ASR 降至 12.5%，优于最强基线 ClawKeeper（14.7%），且在所有三个攻击表面上均达到最低或并列最低 ASR。
2. **良性行为保持优异**：良性通过率 97.4%，人工介入率 0%，延迟仅 0.69 秒/案例，远低于其他需要人工介入或高延迟的方法。
3. **效率-安全权衡更优**：相较于 ClawKeeper（延迟 64.3s, ASR 14.7%），TokenWall 延迟降至 16.9s，ASR 更低，且人工介入率从 69.3% 降至 33.0%。
4. **组件不可或缺**：移除本地审计器会大幅增加大模型调用率和延迟；移除降级仲裁导致 ASR 上升（20.4%）；移除重写增加人工介入率。纯大模型方案延迟过高（82.4s）。

6. **模型规模鲁棒**：即使使用 1.7B 本地模型也能获得 23.1% ASR（远优于规则基线），4B 模型提供了最佳性价比。

## 7. 优点

1. **新颖的防御粒度**：以令牌流为单元，在跨边界传输点进行审计，比传统基于动作或提示的审计更细粒度、更及时。
2. **本地主导、降级为辅的管道**：利用轻量本地模型处理常见案例，仅升级高风险/模糊案例，平衡了安全性与效率。
3. **支持重写**：恢复性重写机制在去除风险的同时尽量保留良性语义，减少不必要的拦截或人工干预。
4. **系统化设计**：预检查、本地审计、降级仲裁三层联动，输出结构化报告指导下游策略，失败闭合设计增强鲁棒性。
5. **实验完备**：详细的消融实验（架构、报告设计、模型规模）和诊断指标（重写效用、不确定性校准）验证了设计选择的合理性。

## 8. 不足与局限

1. **实验覆盖有限**：仅使用 CIK-Bench 作为主要安全基准，该基准针对 OpenClaw 风格的个人代理，不能代表所有持久化代理部署（不同数据源、用户偏好、长期交互模式）。外部泛化性有待验证。
2. **依赖本地模型质量**：方法效果依赖于本地审计器的语义准确性和运行时元数据质量。若本地模型能力不足或元数据不完整，可能导致漏报或误报。文中已指出未来可改进更专门的本地审计器。
3. **不完全解决所有安全问题**：不处理主机被攻破、凭据在代理运行时外被盗、拒绝服务攻击、用户在被准确警告后仍授权有害行为等情况。属于运行时中介而非完整安全方案。
4. **延迟和部署成本依赖实现**：实验中的延迟数据基于特定推理栈，实际部署可能因硬件、缓存策略等不同。文中承认可通过优化本地推理、缓存等进一步降低延迟。
5. **降级仲裁仍需远程调用**：虽然大大减少了远程调用比例，但高风险案例仍需大模型，仍存在潜在隐私和延迟风险（尽管论文已大幅降低）。
6. **重写策略依赖语义分离**：当风险跨度与良性内容不可分离时，重写可能无效，需转人工或拦截，可能导致部分良性功能丧失（实验中重写后仍有33.3%的攻击仍可操作，需后续仲裁）。
