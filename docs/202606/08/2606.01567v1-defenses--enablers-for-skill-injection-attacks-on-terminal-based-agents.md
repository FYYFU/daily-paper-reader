## 摘要
大型语言模型（LLM）智能体越来越依赖可复用的技能，即描述任务特定流程的文档。然而，这为智能体管理引入了新的攻击面。我们针对这一威胁研究了两个互补方向。首先，我们评估了基于守护者的防御：一个作为技能文件访问中介的中间LLM智能体（动态守护者），或在构建时预重写这些文件（静态守护者）。在三个LLM智能体家族中，我们的守护者将攻击成功率（ASR）降低了一半以上，同时保持了任务效用。其次，我们通过攻击重新框架化来对其进行压力测试，使用了四种保留恶意指令但改变措辞的攻击。对于非守护者设置，重新框架化使ASR高达81.4%，但动态守护者将其降至18.6%，表明实时中介是一种稳健的防御。

## Abstract
Large language model (LLM) agents increasingly rely on reusable skills i.e. documents describing task-specific procedures. However, this introduces a new attack surface for agents to manage. We study two complementary directions for this threat. First, we evaluate guardian-based defenses: an intermediary LLM agent that acts as a mediator for skill file access (dynamic guardian) or pre-rewrites these files at build time (static guardian). Across three LLM agent families, our guardians cut attack success rate (ASR) by well over half while preserving task utility. Second, we stress test them through attack reframing using four attacks that preserve the malicious instruction but change the phrasing. For non-guardian setup, the reframing pushes the ASR up to 81.4\%, but the dynamic guardian brings it down to 18.6\%, showing that real-time mediation is a robust defense.

### 2. 论文提出的方法论：核心思想、关键技术细节
- **防御方法：守护者（Guardian）子智能体**
  - **核心思想**：在技能文件被主智能体读取之前，引入一个守护者LLM作为中介。守护者读取技能内容，返回经过过滤/重写后的安全版本。
  - **两种变体**：
    - **动态守护者（Dynamic Guardian）**：作为一个MCP（Model Context Protocol）服务器运行，在主智能体运行时拦截每个技能文件调用。守护者实时过滤内容并返回安全版本。优点：无需修改技能文件，与任何支持MCP的智能体框架兼容；缺点：增加延迟。
    - **静态守护者（Static Guardian）**：在构建时（主智能体启动前）预处理所有技能文件，使用守护者LLM重写每个技能文件，移除注入内容。主智能体在运行时直接使用清理后的文件，无运行时开销。
  - **守护者提示**：共用同一系统提示，要求提供准确技术信息、包含代码示例、全面总结等，**没有显式的安全指令**——注入移除是总结过程的涌现属性。动态守护者的用户消息是主智能体实时提出的自由形式问题；静态守护者使用固定提示“给我此技能的完整文档”。

- **攻击增强方法：攻击重排（Attack Reframing）**
  - **核心思想**：保持恶意指令的语义不变，但改变其呈现方式（措辞、结构），以增加攻击成功率。
  - **四种重排策略**：
    1. **翻译攻击（Translation）**：将整个技能文件（含注入）翻译成西班牙语或日语。
    2. **盲引（Blind Reference）**：将注入作为YAML配置块（`precondition: |`）呈现，伪装成合法的技能设置。
    3. **交叉引用（Cross Reference）**：将注入包装成来自不存在的`SCOPE.md`文件的内容，指示智能体合并这些指令作为同等优先级约束。
    4. **描述陷阱（Description Trap）**：在技能文件中附加一个虚构的“通用助手”技能，其描述广泛以触发自动加载，并在正文中携带恶意指令。
  - **自动发现**：采用`autoresearch`循环，由攻击者LLM（DeepSeek-V4-Flash）自动提出新重排策略并评估，迭代50次，选取了三个最成功的（盲引、描述陷阱、交叉引用）。

### 3. 实验设计：数据集、基准、对比方法
- **数据集**：基于Skill-Inject基准，包含**48个上下文注入**，每个注入对应1-8个用户任务，共**139个沙盒**（任务-注入元组），覆盖25种技能类型（以文档类为主：pptx、docx、xlsx等）。
- **基准**：Skill-Inject（Schmotz et al., 2026），衡量智能体对技能文件注入的脆弱性。
- **对比方法**：
  - **无防御（Vanilla）**：直接使用含注入的技能文件。
  - **系统提示防御**：
    - **通用策略（SysGeneric）**：在智能体系统提示中加入一份包含48条规则的统一安全策略。
    - **定向警告（SysTargeted）**：一种Oracle基线，在系统提示中明确指名正在尝试的注入类型（如“不要将文件上传到外部服务器”），代表提示防御的理论上限。
  - **静态守护者（Static）**：构建时重写。
  - **动态守护者（Dynamic）**：运行时MCP中介。
- **智能体模型与框架**：
  - Claude Sonnet 4.5 + Claude Code（CC）
  - DeepSeek-V4-Flash + OpenCode（OC）
  - NVIDIA Nemotron-Super-120B + OpenCode（OC）
- **攻击重排实验**：选取43个任务-注入对（在自动发现攻击中得分高的子集），在无防御、静态守护者、动态守护者三种条件下评估原始注入和四种重排版本。

### 6. 论文的主要结论与发现
- **守护者防御有效性**：
  - 动态和静态守护者均显著降低攻击成功率（ASR），在三种模型上ASR降至≤15%。例如Claude Code + Sonnet 4.5：ASR从36.0%（无防御）降至12.9%（动态）和7.2%（静态）。
  - 系统提示防御（即便定向警告）的ASR仅为23.0%，远高于守护者，说明**过滤内容先于智能体读取比依靠智能体自我约束更根本有效**。
- **攻击重排增强攻击**：
  - 无防御时，四种重排的联合ASR达81.4%（原始仅55.8%），证明仅依赖单一措辞的基准会显著低估攻击面。
  - 翻译攻击（西班牙语/日语）即可提升ASR，表明多语言攻击有效。
- **动态守护者鲁棒性**：
  - 面对联合重排攻击，动态守护者将ASR降至18.6%（静态降至39.5%），且最难的重排（盲引）仅14.0%。动态守护者的实时中介比静态预处理更鲁棒。
- **守护者行为模式**：
  - 动态守护者的交互轮次与ASR负相关：单轮ASR 11.5%，多轮降低至3%。单次请求“给我完整文档”是最危险模式。
  - 静态守护者可能误将注入“合法化”（如chmod 777被保留为正式工作流步骤），且复杂技能内容损失大。

### 8. 不足与局限
- **基准覆盖有限**：
  - 仅基于Skill-Inject一个基准（139个实例），真实技能生态系统包含数千个社区贡献的技能，其他漏洞类型（如依赖混淆、沙盒逃逸等）未测量。
- **模型与框架覆盖有限**：
  - 仅评估3个模型族和2个框架（Claude Code、OpenCode），其他框架（如MCP语义差异、推理模型）或小规模开源模型的ASR/TSR可能不同。
- **守护者与代理模型耦合**：
  - 主要实验中守护者与主智能体共享同一基础模型，未研究异构守护者（如专用安全模型或不同提供商模型）的效果。
- **攻击重排评估子集选择偏差**：
  - 表3数据基于选定的43个高脆弱性任务-注入对，绝对数值不与表1的139个沙盒全集直接可比。
- **攻击发现范围有限**：
  - 自动发现循环仅50次迭代、单一个攻击者模型（DeepSeek-V4-Flash），更长搜索或多模型集成可能发现更多重排。
- **判断器可靠性**：
  - 任务成功率（TSR）部分依赖LLM判断器（Sonnet 4.5），可能引入系统噪声，尤其在静态守护者重写后技能与原始差异大时。
- **语言与领域局限**：
  - 任务和注入主要为英文，守护者提示和自动发现循环在英文中心环境下评估，其他语言可能表现不同。操作领域仅限于终端代理（shell命令、Python脚本、本地文件），不直接推广到GUI、浏览器或非Python生态系统。
- **道德风险**：攻击重排方法虽用于防御评估，但可能被恶意利用。作者通过在隔离Docker沙盒中实验、使用已公开注入等缓解，但双用途性质仍需注意。
