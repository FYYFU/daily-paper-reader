## 摘要
像OpenClaw和Nanobot这样的本地LLM智能体在最终用户机器上运行，并通过自然语言目标作用于主机资源——shell、文件系统、浏览器、存储的凭据和消息应用程序。这些智能体已成为特权软件运行时，在用户意图、模型输出和主机级操作之间进行中介。现有研究通过提示注入、恶意技能、市场风险或智能体黑盒评估来描述这一领域。但执行这种中介的实现层——提示构建器、解析器、工具调度器、技能加载器、内存写入器、网络客户端和权限门——仍然是一个未经过审查的安全边界。据我们所知，之前的工作没有审查智能体的源代码树来审计这些组件的实现级安全弱点。我们提出了CLAWAUDIT，一个用于衡量本地LLM智能体运行时中漏洞暴露程度的静态审计框架。CLAWAUDIT从STRIDE推导出一个五类漏洞分类法，并开发了针对智能体特定模式的自定义静态分析规则，这些模式在现有的漏洞分析规则集中缺失。我们在两个后端中实例化了该分类法：47条Semgrep YAML规则和30条CodeQL查询，并在OPENCLAWBENCH上进行了评估，该基准包含来自OpenClaw仓库的446个源代码级公告，并按时间分为229个规则推导（训练）公告和217个保留（测试）公告。在保留测试中，CLAWAUDIT将Semgrep的召回率从21.7%（Pro基线）提升到66.8%，将CodeQL的召回率从13.8%（安全扩展）提升到75.1%。对于所有四种配置，训练/测试差距保持在4个百分点以内，表明这些规则能够泛化到规则编写期间未见过的漏洞。初步的实时代码审计表明，这些面向召回率的规则需要手动分类，这促使在生产部署前进行语义过滤。

## Abstract
Local LLM agents such as OpenClaw and Nanobot run on end-user machines and act on host resources - the shell, filesystem, browser, stored credentials, and messaging applications - through natural-language goals. These agents have become privileged software runtimes that mediate between user intent, model outputs, and host-level actions. Existing research characterizes the landscape through prompt injection, malicious skills, marketplace risks, or black-box evaluation of agents. But the implementation layer that performs this mediation, the prompt builder, parser, tool dispatcher, skill loader, memory writer, network client, and permission gate, has remained an unexamined safety boundary. To our knowledge, no prior work has examined the agent's source tree to audit these components for implementation-level security weaknesses. We present CLAWAUDIT, a static auditing framework for measuring vulnerability exposure in local LLM agent runtimes. CLAWAUDIT derives a five-category vulnerability taxonomy from STRIDE and develops custom static-analysis rules that target agent-specific patterns absent from established rule sets for vulnerability analysis. We instantiate the taxonomy in two backends, 47 Semgrep YAML rules and 30 CodeQL queries, and evaluate on OPENCLAWBENCH, a benchmark of 446 source-code-level advisories from the OpenClaw repository and split temporally into 229 rule-derivation (train) and 217 held-out (test) advisories. On the held-out test, CLAWAUDIT raises Semgrep recall from 21.7% (Pro baseline) to 66.8%, and CodeQL recall from 13.8% (security-extended) to 75.1%. Train/test gaps remain within 4 percentage points for all four configurations, indicating that the rules generalize to vulnerabilities unseen during rule writing. A preliminary live-code audit shows that these recall-oriented rules require manual triage, motivating semantic filtering before production deployment.

- **整体含义**：论文首次提出对本地LLM智能体运行时层进行实现级漏洞审计，指出该层是一个未被充分研究的安全边界，并开发了系统化的静态审计框架来弥补这一缺口。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：基于STRIDE威胁模型推导出针对本地智能体运行时层的五类漏洞分类法，然后针对每一类开发自定义静态分析规则，并分别在Semgrep（语法模板匹配）和CodeQL（关系图查询）两个后端中实现。
- **关键技术细节**：
  - 从STRIDE的六个威胁类别中排除DoS（运行时资源耗尽不可静态捕获）和Repudiation（审计日志完整性，与实现弱点正交），保留Tampering、Elevation of Privilege、Information Disclosure、Spoofing，并将Tampering拆分为CAT-1（提示处理）和CAT-2（工具/技能执行），共得到五类：
    - CAT-1：提示处理（Prompt Handling）——未经过滤的文本进入提示构建或内存。
    - CAT-2：工具/技能执行（Tool/Skill Execution）——模型或用户控制的数据影响文件系统、shell或技能加载。
    - CAT-3：权限与隔离（Permission & Isolation）——未能隔离工具、技能或执行上下文。
    - CAT-4：网络与通信（Network & Communication）——不安全地构造请求、日志或外部交互。
    - CAT-5：认证与授权（Authentication & Authorization）——未能强制执行用户、会话或资源的授权。
  - 针对每个类别，分析OpenClaw已披露公告中的脆弱代码模式，编写47条Semgrep YAML规则和30条CodeQL查询。规则不仅检测漏洞模式，还检查缺失的防护（如缺少消毒函数、缺少路径包含检查）。
  - 使用时间分割的基准（OPENCLAWBENCH）进行验证：训练集229个公告（规则推导），测试集217个公告（保留，规则定稿后才收集）。

## 6. 论文的主要结论与发现

1. **通用静态分析工具对本地智能体运行时漏洞检测率很低**：Semgrep Pro在测试集上召回率仅21.7%，CodeQL安全扩展仅13.8%，并集仅31.3%。原因在于现有规则针对传统Web应用，而非智能体特有的操作（如提示构建、技能加载等）。
2. **CLAWAUDIT大幅提升召回率**：在测试集上，Semgrep+CLAWAUDIT召回率提升至66.8%（+45.1 pp），CodeQL+CLAWAUDIT提升至75.1%（+61.3 pp）。提升覆盖所有严重等级和几乎所有CWE类别，尤其对符号链接跟随（CWE-59，从0%升至86.7%）、路径遍历（从16%升至80%+）、TOCTOU（从10-30%升至90%）、命令注入（提升30-60 pp）等效果显著。
3. **规则能泛化到未见过的公告**：训练/测试召回率差距均在±4 pp内，说明规则捕捉的是重复出现的实现结构，而非记忆训练集。
4. **当前规则是召回优先，实时代码精度低**：在HEAD代码上，每个后端产生超过3000个发现，手动检查25个中发现仅3个真阳性（精度12%），说明需要后续语义过滤或优先级排序才能实用。
5. **规则检测对语义定义的弱点（如不完整黑名单、信息泄露）效果有限**：CWE-184、CWE-200、CWE-639等仍存在较大残留缺口，因为脆弱代码在语法上与正确代码无异，需要外部策略规范或语义推理。

## 7. 优点：方法或实验设计上的亮点

- **首次对智能体运行时进行系统化实现级审计**，填补了安全研究空白。
- **基于STRIDE威胁模型推导分类**，具有理论支撑和可解释性，而非临时枚举。
- **在两个不同的静态分析后端（Semgrep和CodeQL）上实现并评估**，提供了跨工具的可比性，并揭示了引擎差异对召回的影响（如CodeQL更好的过程间缺失防护检查）。
- **构建了专门针对智能体运行时的基准（OPENCLAWBENCH）**，包含446个真实历史公告，并通过时间分割严谨地评估泛化能力。
- **公开了所有规则和基准（GitHub仓库）**，便于复现和后续研究。
- 明确指出了规则检测的边界（语义定义类漏洞），并提出了未来改进方向（如智能体特定的污点分析、策略挖掘、LLM辅助规则合成、运行时插桩）。

## 8. 不足与局限

- **仅针对OpenClaw**，未在其他本地LLM智能体（如Nanobot、PicoClaw）上验证泛化性。
- **基准只包含已公开披露的漏洞**，未覆盖未报告或尚未被认定为安全问题的漏洞类。
- **基准是正样本密集的，无法单独衡量假阳性率**；精度评估仅50个样本，统计效力有限。
- **规则精度低（12%）**，在大型代码库上产生大量假阳性，与生产部署的要求相差较远。
- **规则无法检测语义依赖的漏洞**（如不完整黑名单、信息泄露、资源所有权授权），需要更高级的语义分析。
- 分类法基于OpenClaw当前观察到的运行时结构，若出现新组件（如多智能体协调器）可能需要调整。
- 实验未涉及运行时行为依赖的漏洞（如竞争条件仅通过模式匹配可能不够，TOCTOU虽被覆盖但依赖于结构模式）。
- 未公开提及实验的重复运行次数或随机性控制细节（如Semgrep/CodeQL版本固定，但未说明内部随机性）。
