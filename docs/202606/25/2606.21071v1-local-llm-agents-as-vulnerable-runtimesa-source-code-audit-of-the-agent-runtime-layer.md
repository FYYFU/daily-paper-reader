## 摘要
诸如OpenClaw和Nanobot之类的本地LLM代理在最终用户机器上运行，并通过自然语言目标对主机资源（如shell、文件系统、浏览器、存储的凭据和消息应用程序）进行操作。这些代理已成为特权软件运行时，在用户意图、模型输出和主机级操作之间进行中介。现有研究通过提示注入、恶意技能、市场风险或黑盒评估代理来描述这一领域。但执行这种中介的实现层，即提示构建器、解析器、工具调度器、技能加载器、内存写入器、网络客户端和权限门，仍然是一个未经检验的安全边界。据我们所知，之前没有工作检查代理的源代码树来审计这些组件的实现级安全弱点。我们提出了CLAWAUDIT，一个用于衡量本地LLM代理运行时漏洞暴露的静态审计框架。CLAWAUDIT从STRIDE导出一个五类漏洞分类，并开发针对代理特定模式的自定义静态分析规则，这些模式在现有的漏洞分析规则集中不存在。我们将该分类实例化为两个后端：47条Semgrep YAML规则和30条CodeQL查询，并在OPENCLAWBENCH基准上进行评估，该基准包含来自OpenClaw仓库的446条源代码级建议，并按时间分为229条规则推导（训练）和217条保留（测试）建议。在保留测试上，CLAWAUDIT将Semgrep召回率从21.7%（Pro基线）提升到66.8%，CodeQL召回率从13.8%（安全扩展）提升到75.1%。所有四种配置的训练/测试差距保持在4个百分点以内，表明这些规则能够泛化到规则编写期间未见过的漏洞。初步的实时代码审计表明，这些面向召回率的规则需要手动分类，这促使在生产部署前进行语义过滤。

## Abstract
Local LLM agents such as OpenClaw and Nanobot run on end-user machines and act on host resources - the shell, filesystem, browser, stored credentials, and messaging applications - through natural-language goals. These agents have become privileged software runtimes that mediate between user intent, model outputs, and host-level actions. Existing research characterizes the landscape through prompt injection, malicious skills, marketplace risks, or black-box evaluation of agents. But the implementation layer that performs this mediation, the prompt builder, parser, tool dispatcher, skill loader, memory writer, network client, and permission gate, has remained an unexamined safety boundary. To our knowledge, no prior work has examined the agent's source tree to audit these components for implementation-level security weaknesses. We present CLAWAUDIT, a static auditing framework for measuring vulnerability exposure in local LLM agent runtimes. CLAWAUDIT derives a five-category vulnerability taxonomy from STRIDE and develops custom static-analysis rules that target agent-specific patterns absent from established rule sets for vulnerability analysis. We instantiate the taxonomy in two backends, 47 Semgrep YAML rules and 30 CodeQL queries, and evaluate on OPENCLAWBENCH, a benchmark of 446 source-code-level advisories from the OpenClaw repository and split temporally into 229 rule-derivation (train) and 217 held-out (test) advisories. On the held-out test, CLAWAUDIT raises Semgrep recall from 21.7% (Pro baseline) to 66.8%, and CodeQL recall from 13.8% (security-extended) to 75.1%. Train/test gaps remain within 4 percentage points for all four configurations, indicating that the rules generalize to vulnerabilities unseen during rule writing. A preliminary live-code audit shows that these recall-oriented rules require manual triage, motivating semantic filtering before production deployment.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：本地LLM代理（如OpenClaw、Nanobot）直接运行在用户机器上，通过自然语言目标操纵shell、文件系统、浏览器、存储凭据和消息应用等主机资源。这些代理实际上已成为一种“特权软件运行时”，在用户意图、模型输出和主机操作之间进行中介。然而，现有关于LLM代理安全的研究主要集中在提示注入、恶意技能、市场风险或黑盒行为评估上，从未系统地打开代理本身的源代码树，对其实现层（提示构建器、解析器、工具调度器、技能加载器、内存写入器、网络客户端和权限门）进行源码级安全审计。
- **核心问题**：代理运行时层的实现缺陷是一个未经检验的安全边界。这些组件中的任何一个出错，都可能导致从不受信任输入直达主机上特权操作的路径。
- **整体含义**：本地LLM代理的安全性不仅取决于模型对齐或技能审查，还取决于其运行时源代码是否正确、安全。本文首次系统性地对代理运行时层进行源代码审计，填补了该空白。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：通过威胁建模（STRIDE）指导，推导出代理运行时漏洞的五类分类；然后针对每类漏洞，开发领域特定的静态分析规则（支持两个后端：Semgrep和CodeQL），用于检测代理运行时中的实现级弱点。
- **关键技术细节**：
  - **漏洞分类（Taxonomy）**：从STRIDE威胁模型出发，结合代理运行时的数据流图，识别出三个信任边界，最终得到五类漏洞：
    - CAT-1: 提示处理（Prompt Handling）——未受信任文本进入提示构建或内存。
    - CAT-2: 工具/技能执行（Tool/Skill Execution）——模型或用户控制的参数影响文件系统、shell或技能加载。
    - CAT-3: 权限与隔离（Permission & Isolation）——运行时未能隔离工具、技能或执行上下文。
    - CAT-4: 网络与通信（Network & Communication）——运行时以不安全方式构建请求、日志或外部交互。
    - CAT-5: 认证与授权（Authentication & Authorization）——运行时未能对用户、会话或资源强制执行授权。
  - **规则构建**：针对每类漏洞，从已公开的OpenClaw安全公告（GHSA）中识别可重复的实现结构，然后分别编写Semgrep YAML规则（47条）和CodeQL查询（30条）。规则编码了代理特有的人名模式、缺少防护检查等逻辑。
  - **静态分析后端**：Semgrep基于AST语法模板匹配，适合密集模式枚举；CodeQL通过声明式谓词在程序关系图上进行有限的过程间数据流分析。两个后端共享相同的分类和目标模式。
  - **检测与归属**：原始发现（文件、行号、代码片段）通过文件级归属判断是否检测到公告：如果发现在补丁提交修改的文件内，则认为检测到该公告。

## 3. 实验设计：数据集、基准、对比方法

- **数据集**：构建了 **OPENCLAWBENCH** 基准，包含从OpenClaw仓库公开发布的446个GitHub安全公告（GHSA），时间覆盖2026-01-31至2026-05-15。
  - 按发布时间划分为规则推导集（训练集，229个公告，2026-04-01前）和保留测试集（测试集，217个公告，2026-04-01及之后）。规则开发仅使用训练集，测试集在规则最终确定后收集且仅评估一次。
  - 每个公告关联了CWE标签、严重性等级、修复提交、受影响文件等元数据。
  - 支持84个不同的CWE类，其中最多的包括：CWE-863授权绕过（77）、CWE-22路径遍历（31）、CWE-78命令注入（30）、CWE-918 SSRF（20）等。
- **基准与对比方法**：
  - **Semgrep基线**：Semgrep Pro默认自动配置（约2,815条内置+社区规则）。
  - **CodeQL基线**：`javascript-security-extended` 查询套件（104条内置查询）。
  - **CLAWAUDIT配置**：在每个基线之上添加CLAWAUDIT规则（Semgrep + 47条规则 / CodeQL + 30条查询）。
- **评估指标**：主要指标是公告级召回率，即被配置检测到的公告比例。另外对实时仓库HEAD版本进行了精度抽样审计。

## 4. 资源与算力

- 论文中未使用GPU或进行模型训练。所有静态分析实验在一台Linux x86_64主机上运行，配备64逻辑核心、256GB系统内存和本地NVMe存储。主机在扫描期间空闲。
- **扫描时间预算**：
  - 每个公告快照的Semgrep扫描：默认线程，120秒超时。
  - 每个公告快照的CodeQL扫描：构建单快照数据库，分析使用 `-threads=2`，每个快照限时1小时。
  - HEAD实时扫描（16,433文件）：构建单个CodeQL数据库，每个查询限时4小时；超过4小时未完成的查询记为不收敛（实际有21条基线查询未收敛，CLAWAUDIT所有查询均在数秒至数分钟内完成）。
- 未提供显式GPU或训练算力要求，因为该方法完全是静态代码分析。

  - 对比了当前最先进的通用静态分析工具（Semgrep Pro、CodeQL security-extended）并报告其召回率，基线选择合理、公正。

## 6. 论文的主要结论与发现

1. **通用静态分析工具检测能力低**：Semgrep Pro在保留测试集上召回率仅21.7%，CodeQL security-extended仅13.8%。两者并集也仅31.3%。表明现有常规Web应用规则集无法覆盖代理运行时特有的漏洞模式。
2. **CLAWAUDIT显著提升召回率**：
   - Semgrep+CLAWAUDIT：训练集65.5%（提升40.2pp）、测试集66.8%（提升45.1pp）。
   - CodeQL+CLAWAUDIT：训练集79.0%（提升66.8pp）、测试集75.1%（提升61.3pp）。
   - 提升在所有严重性等级和多数CWE类上都一致，尤其对symlink following（0%→86.7%）、path traversal（16%→84%）、TOCTOU（10%→90%）等提升最大。
3. **规则泛化到未见漏洞**：保留测试集与规则推导集的召回率差距均在±4pp以内，表明规则捕获了递归性实现结构，而非记忆训练样本。
4. **当前规则召回率高但精确度低**：对HEAD实时代码，Semgrep+CLAWAUDIT产生3,473个发现，CodeQL+CLAWAUDIT产生3,756个发现。抽样精度审计显示两后端模式级精确度均为12%（50个发现中仅6个真阳）。需要手动分类过滤大量误报。
5. **规则检测的边界**：对具有重复代码形状的漏洞（如路径遍历、符号链接跟随、命令注入、SSRF、TOCTOU、缺少局部防护）效果好；对语义定义的漏洞（如不完整黑名单CWE-184、信息暴露CWE-200、资源所有权授权CWE-639）效果差，需要外部规范推断。
6. **启示**：提示级防御（如防止提示注入）是必要的但不充分；市场技能审计不能替代运行时审计；特权运行时视角可推广到其他本地LLM代理。

## 7. 优点

- **新颖性**：首次对本地LLM代理运行时层进行系统性的源代码级安全审计，填补了代理安全研究在实现层面上的空白。
- **方法论系统**：基于STRIDE威胁模型推导漏洞分类，而非直觉枚举，保证了覆盖度和互斥性。利用信任边界确定规则归属，分类清晰且可操作。
- **双后端实现**：在同一分类下分别在Semgrep和CodeQL上实现规则，可比较两种分析引擎的优势与局限，增强了方法通用性。
- **构建了专门的基准**：OPENCLAWBENCH（446个真实已披露公告）采用时间划分，评估真实部署场景下的泛化能力。公开可用，促进后续研究。
- **显著的召回率提升**：从基线约20%提升至70%左右，量化了通用规则集的不足，并证明领域特定静态分析的有效性。
- **诚实报告局限性**：明确指出了召回率导向、精确度低、仅针对OpenClaw、依赖公开公告等局限，并提出了未来改进方向（语义过滤、过程间污点分析、策略挖掘等）。
- **负责任披露**：对于HEAD实时代码中发现的新问题，遵循负责任披露流程。

## 8. 不足与局限

- **覆盖范围有限**：仅针对OpenClaw一个代理进行实验，声称可推广到同类架构但未在Nanobot等上验证。未来需要跨代理泛化研究。
- **基准仅依赖公开公告**：未包含未报告漏洞或未识别为安全问题的bug，可能会高估检测率。且基准是召回率导向的，无法直接衡量误报成本。
- **精确度极低**：HEAD实时代码的精度仅12%，大量误报，部署需大量人工筛选，不适合直接作为生产扫描器。
- **精度审计样本量小**：仅50个发现、单审查者，可能不够稳定，无法细致估计每规则精度方差。
- **对语义型CWE效果差**：对不完整黑名单、信息暴露、资源所有权授权等依赖外部意图的漏洞，静态规则基本无能为力，需要语义分析或策略挖掘补充。
- **未覆盖所有运行时组件**：分类基于当前观察到的OpenClaw运行时结构，随着新组件（如多代理协调器、持久化内存后端）出现可能需要扩展。
- **过程间分析能力有限**：虽然CodeQL支持有限的过程间查询，但大多规则仍局限在函数内，未实现全局污点分析。基线中21条CodeQL污点查询在HEAD仓库上未在4小时内收敛，表明扩展存在困难。
- **缺乏消融实验**：没有系统分析不同规则子集对召回率的贡献，难以判断哪些规则是最有效的。
- **可重复性**：虽然公开了规则和基准清单，但第三方重现仍依赖GHSA API和OSV API的状态，可能随时间变化。
