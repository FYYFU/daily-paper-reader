## 摘要
基于LLM的护栏已作为一种高效防御手段，用于抵御自主智能体中的提示注入和越狱攻击。然而，我们发现正是这种实现防护的推理和任务跟随能力引入了一种新的漏洞：攻击者可以注入精心构造的数据，将护栏困在扩展的推理循环中，从而实施系统性的拒绝服务攻击。为系统性地揭示这一威胁，我们设计了一个束搜索优化框架，利用由策略库引导的LLM提出器，构造自然语言有效载荷以最大化护栏推理长度。基于对护栏遵循模式特性的观察，我们还提供了另一种由机制感知的结构突变驱动的攻击框架，其计算负载更低。攻击效果分两部分系统评估。首先，在独立评估中，攻击可泛化至多种护栏架构、安全模板和智能体基准。在单个开源代理上优化的有效载荷成功迁移至八个领先模型骨干（如Claude、GPT、Gemini、DeepSeek和Qwen），实现了13至63倍的令牌放大。其次，在端到端的真实世界智能体部署（网页、桌面、代码和多智能体系统）中，攻击揭示了高达148倍的延迟放大。我们表明，单个被毒化的文档即可饱和共享护栏基础设施，有效饿死共置智能体并使整个系统瘫痪。通过揭示这一可用性缺陷，我们的工作强调了开发成本受限、推理鲁棒的护栏的迫切需求。

## Abstract
LLM-based guardrails have emerged as a highly effective defense against prompt injection and jailbreak attacks in autonomous agents. However, we reveal that the very reasoning and task-following capabilities enabling this protection introduce a novel vulnerability: attackers can inject crafted data to trap the guardrail in extended reasoning loops, effectuating a systematic denial-of-service (DoS) attack. To systematically expose this threat, we design a beam-search optimization framework that crafts natural-language payloads to maximize guardrail reasoning length, utilizing an LLM proposer guided by a strategy bank. Based on the observation of guardrail's schema-following nature, we also provide another attack framework driven by mechanism-aware structural mutations with less computational load. The attack efficacy is systematically evaluated in two parts. First, in standalone evaluations, the attack generalizes across diverse guardrail architectures, safety templates, and agent benchmarks. Payloads optimized on a single open-source surrogate successfully transfer to eight leading model backbones (e.g., Claude, GPT, Gemini, DeepSeek, and Qwen), achieving a 13--63$\times$ token amplification. Second, in end-to-end real-world agent deployments (web, desktop, code, and multi-agent systems), the attack reveals up to a 148$\times$ latency amplification. We show that a single poisoned document can saturate shared guardrail infrastructures, effectively starving co-located agents and paralyzing the entire system. By uncovering this availability flaw, our work underscores the urgent need to develop cost-bounded, reasoning-robust guardrails.

## 1. 核心问题与整体含义（研究动机和背景）
- **背景**：LLM-based autonomous agents（如web agents、desktop agents、code agents、multi-agent systems）正广泛应用于真实场景。这些agent需要读取第三方内容（网页、文件、API响应等），其安全性至关重要。传统轻量级过滤器（规则或分类器）无法应对复杂的上下文威胁（如间接提示注入、越狱攻击）。因此，业界转向**LLM-based guardrails**：在agent执行每个动作前，由一个LLM模型对完整上下文（轨迹+动作）进行推理，输出结构化安全裁决（allow/block）。这种范式在AgentDojo、ToolSafe、NeMo Guardrails、OpenHands等系统中成为主流。
- **核心问题**：论文揭示了一个此前被忽视的漏洞——**reasoning-extension DoS（推理扩展型拒绝服务攻击）**。LLM-based guardrail的推理能力本用于安全分析，但攻击者可以通过在被agent获取的外部内容中注入构造的文本，使guardrail将其视为自身的分析模板（schema），从而陷入无限扩展的推理循环，消耗大量计算资源。由于guardrail位于每个agent动作的**关键路径**上，这种攻击直接导致服务可用性丧失，甚至通过共享基础设施瘫痪整个系统。
- **整体含义**：guardrail作为安全防线本身成为了攻击目标。该漏洞利用的是guardrail最核心的优势——遵循结构化模板进行推理的能力。这揭示了“安全推理能力”与“可用性”之间的固有矛盾，亟需研发成本受限、推理鲁棒的新型护栏机制。

## 2. 论文提出的方法论：核心思想、关键技术细节
### 核心思想
- 现有DoS攻击（如Off-task decoy、优化型对抗字符串、工具链调用）尝试让LLM偏离任务，但guardrail被严格任务限定（安全分析），难以被干扰。论文的**关键洞察**是：**不在欺骗guardrail离开任务，而是在任务内部延长其推理过程**。
- guardrail具有“schema-following”（遵循分析模板）的特性。攻击者注入的文本如果模仿guardrail自身的分析结构（如风险分类、评估指标、反捷径条款），guardrail会将其解释为需要处理的分析内容，从而机械地填充模板，产生超长输出。
- 通过量化分析发现，这种循环有特征：注意力循环（响应token对方案头部的注意力权重比基线高9.6倍）、熵崩塌（生成时每token熵从0.264比特降至0.132比特），表明模型进入模板填充模式而非真正推理。

### 技术细节：束搜索优化框架
论文设计了一个通用的束搜索优化框架，以最大化guardrail推理长度为目标。公式化：
```
p* = argmax_{p∈P} E_{c~C}[ |Φ(G_s(T(c[p])))| ]
```
其中`G_s`是代理guardrail，`T`是模板，`c[p]`是注入payload后的上下文，`Φ`提取推理部分长度。

框架维护一个大小为k的候选payload束，每轮每个父payload产生b个子变异，用适应度函数评分后选择top-k，并确保多样性（通过序列相似度阈值δ）。早期停止条件为连续P次无提升。

### 两种具体实例化
- **实例化I：LLM-as-Proposer（LLM作为提议器）**
  - 使用一个LLM（如GPT-5.2）作为提议器，接收父payload、guardrail的完整推理输出、以及来自**策略库**的指导（策略库累积有效模式并更新平均得分）。
  - 提议器生成b个变异子，保留高分结构。策略库在迭代中可积累100+个不同模式。
  - 优点：无需先验知识，能从零探索自然语言结构空间。

- **实例化II：Mechanism-Aware Optimization（机制感知优化）**
  - 将payload分解为模板插槽（如风险类别列表、枚举深度、反捷径条款、强制标定等），并定义轻量级结构变异算子（类别扩展、深度调整、节重排、反捷径强度增加等）。
  - 适应度函数：`F_mech(p) = F_base(p) + ω·A(p)`，其中`A(p)`是注意力循环分数（通过前向hook计算响应token对方案头部的平均注意力）。
  - 加入熵过滤：早期高熵的候选被拒绝（因为低熵是持续的模板填充标志）。计算开销更低，无需LLM提议器。

### 场景适应性
- 框架可适配不同注入表面（隐藏DOM、代码注释、agent间消息等）和不同guardrail架构，通过替换代理模型和调整变异操作实现。

## 6. 论文的主要结论与发现
- **攻击有效且泛化**：单个开放源码代理模型优化的payload能成功转移至Claude、GPT、Gemini、DeepSeek、Qwen等8个模型，实现13–63× token放大（独立评估）和最高148×延迟放大（端到端）。
- **实际影响严重**：
  - Code Agent：README注入平均25.9×，导致单任务安全开销从~138秒增至59分钟。
  - Multi-agent：单个毒化文档在fan-out、sequential、supervisor架构中造成级联延迟，共享guardrail引发头行阻塞，共置agent吞吐量下降23.3%。
  - Web Agent：隐藏DOM注入在checkout任务中达到131×延迟放大，超时策略导致安全与可用性困境（fail-open允许未审查交易，fail-closed阻止合法操作）。
  - Desktop Agent：ShieldAgent的三重检查架构使单动作放大18.42×，且攻击在对话中持续影响后续动作。
- **现有防线无效**：
  - 预推理过滤器（DeBERTa注入检测）无法检测流利的自然语言payload。
  - 硬token预算截断只转移失败模式：fail-open导致安全绕过，fail-closed导致DoS。
  - 更强的推理模型反而更顺从schema，产生更长循环。
- **根本原因**：攻击利用的是guardrail的核心能力——指令遵循和结构化分析，这是其有效性的来源，也是脆弱性的根。

## 7. 优点
- **新颖性和重要性**：首次系统性识别并形式化LLM-based guardrail的可用性漏洞。该威胁此前未被关注，且直接针对安全层本身，具有高实际风险。
- **方法设计巧妙**：
  - 从“使模型更做它想做的事”的角度攻击，而非干扰任务，这在LLM DoS中是一个新思路。
  - 束搜索框架具有通用性，两种实例化了平衡探索成本与效果（LLM-as-Proposer用于发现，Mechanism-Aware用于高效优化）。
  - 策略库和注意力引导评分机制提升了搜索效率和可解释性。
- **实验充分且系统**：
  - 覆盖4种真实agent场景（代码、多agent、Web、桌面），每个场景都有适配说明和端到端量化。
  - 对比多个强baseline，并展示转移性和消融分析，验证了结构元素的关键作用。
- **实际威胁验证**：展示了从单文档毒化到整个系统瘫痪的完整攻击链（资源耗尽 → 共享基础设施饥饿 → 安全绕过），并提供明确量化数据（如148×延迟、23.3%吞吐量下降）。

## 8. 不足与局限
- **攻击前提要求**：攻击者需要能够将payload注入agent未来会获取的外部内容（网页、仓库文件等）。这在现实信任模型中常见（如公开网站、GitHub仓库），但并非所有场景都可行（如企业内网、受控API）。
- **优化成本**：LLM-as-Proposer依赖商业LLM（GPT-5.2）产生变异，存在API成本；Mechanism-Aware仍需训练注意力评分（但开销较小）。对于资源不足的攻击者，可能需要更轻量方案。
- **泛化边界**：实验模型虽覆盖主流，但未测试所有guardrail（如LlamaGuard 3、NeMo Guardrails的其他配置等）。结果在分类器式guardrail（如ShieldAgent）上仍需依赖generation cap绕过，攻击效果可能因模型微调不同而有所变化。
- **防护方案的不足**：论文提出几种防御并论证其失败，但未提出可行缓解方案（仅建议研发成本受限推理鲁棒护栏）。这留给未来工作，当前缺乏实际部署中的缓解措施。
- **未考虑对抗性训练**：如果guardrail经过针对schema攻击的对抗训练（如强化学习抑制模板填充），攻击可能部分失效。论文未验证这点。
- **统计显著性**：端到端实验中任务数量有限（如20个SWE-bench实例），环境可变性可能导致结果波动。但多个任务和表面的平均结果增强了信心。
- **其他潜在漏洞**：论文专注于DoS，但提到可能的安全绕过（fail-open导致未审查动作）。未深入探索DoS-to-bypass的转换或与其他攻击（如提示注入）的组合效应。
