## 摘要
大型语言模型（LLM）越来越多地用于代码生成，引发了它们可能被滥用以生成恶意代码的担忧。与此同时，语法约束解码（GCD）已广泛采用，通过强制语法有效性来提高LLM生成代码的可靠性。在本文中，我们揭示了一个违反直觉的风险：这种面向可靠性的技术本身可能成为一个攻击面。我们发现了一种新的越狱攻击，称为CodeSpear，它利用GCD诱导LLM生成恶意代码。我们的实验表明，仅应用良性的代码语法约束即可有效越狱LLM。为了解决这一漏洞，我们提出了CodeShield，一种安全对齐方法，即使在攻击者控制的语法约束下也能稳健地保持安全行为。CodeShield通过在代码模态中对齐模型，教它在GCD下生成蜜罐代码。这种代码在语义上是无害的，因此不会实现恶意请求，并且在结构上多样化，因此难以通过语法收紧来抑制。同时，当自然语言可用时，CodeShield仍然保留自然语言拒绝。在4个基准测试中对10个流行LLM的实验表明，CodeSpear优于代表性的越狱基线，平均攻击成功率提高了30个百分点以上。CodeShield在CodeSpear下恢复了安全性，同时保持了良性实用性。我们的发现揭示了GCD的一个基本风险，并呼吁对其潜在安全影响给予更多关注。

## Abstract
Large Language Models (LLMs) are increasingly used for code generation, raising concerns that they may be misused to produce malicious code. Meanwhile, Grammar-Constrained Decoding (GCD) has been widely adopted to improve the reliability of LLM-generated code by enforcing syntactic validity. In this paper, we reveal a counterintuitive risk: this reliability-oriented technique can itself become an attack surface. We uncover a new jailbreak attack, termed CodeSpear, that exploits GCD to induce LLMs into generating malicious code. Our experiments show that simply applying a benign code grammar constraint can effectively jailbreak LLMs.   To address this vulnerability, we propose CodeShield, a safety alignment approach that robustly preserves safe behavior even under attacker-controlled grammar constraints. CodeShield aligns the model in the code modality by teaching it to generate honeypot code under GCD. Such code is semantically harmless, so it does not implement the malicious request, and structurally diverse, so it is difficult to suppress through grammar tightening. At the same time, CodeShield still preserves natural-language refusals when natural language is available. Experiments on 10 popular LLMs across 4 benchmarks show that CodeSpear outperforms representative jailbreak baselines and increases the attack success rate by more than 30 percentage points on average. CodeShield also restores safety under CodeSpear while preserving benign utility. Our findings reveal a fundamental risk of GCD and call for greater attention to its potential security implications.

## 2. 论文提出的方法论

### 核心思想
- **攻击（CodeSpear）**：利用GCD将输出空间从 `V*` 收缩到 `L(G)`（语法G定义的语言）。对于恶意提示 `p`，模型在原空间下本应输出自然语言拒绝（如“I cannot assist with that”），但这些拒绝不在 `L(G)` 中，概率为零。模型被迫在代码空间中继续生成，而该空间下模型未经安全对齐，容易产生有害代码。
- **防御（CodeShield）**：通过DPO（直接偏好优化）训练模型，使其在代码模态下优先生成**蜜罐代码**（honeypot code）。蜜罐代码有两个特性：
  - **语义无害**：不实现恶意请求的功能；
  - **结构多样**：从大型代码语料库随机采样，覆盖多种语法结构，难以被攻击者通过收紧语法来排除。

### 关键技术细节
**CodeSpear算法流程**（Algorithm 1）：
1. 初始化前缀 y<1 = ∅。
2. 对于 t = 1 到 T_max：

   - 从重归一化分布 P^G_M(·|p, y<t) 采样 y_t；

3. 若遇到结束符则停止。

该算法不修改模型参数，不依赖梯度，只需调用标准GCD接口。

**CodeShield训练过程**（Algorithm 2）：
- 对每个恶意提示 p：
  - 收集有害代码 y_harmful：用目标模型 M 在 GCD 下采样得到。

  - (p, y_refuse, y^(k)_honeypot) —— 自然语言可用时拒绝优于蜜罐代码；
  - (p, y^(k)_honeypot, y_harmful) —— 代码模态下蜜罐代码优于有害代码。

### 公式说明（文字版）
- GCD生成分布：`P^G_M(y|p) = I[y in L(G)] * ∏ P^G_M(y_t|p, y<t)`，只有语法内的序列概率非零。
- DPO损失：`L_DPO = -E[log σ(s(p,y+) - s(p,y-))]`，其中 `s(p,y) = β log (P_Mθ(y|p) / P_Mref(y|p))`。

## 6. 论文的主要结论与发现

1. **CodeSpear攻击有效**：在10个LLM上，平均ASR从54.92%提升至81.82%（本地），从22.00%提升至67.39%（API），提升超过30个百分点。优于所有基线。
2. **安全对齐在代码模态下缺失**：标准自然语言对齐在GCD下失效。即使模型学会输出`pass`等固定模式，攻击者通过收紧语法（禁用`pass`）可进一步提高ASR（GPT-5从55.49%到70.30%）。
3. **CodeShield有效防御**：在三个模型上，CodeSpear下ASR从83%+降低到5%-8%左右，同时保持正常拒绝能力。比Safe-DPO显著更优（Safe-DPO在GCD下ASR仍高达77%）。
4. **效用保留**：CodeShield对pass@1/3的影响有限，部分指标甚至略有提升。
5. **自适应攻击下依然稳健**：结构化多样性使得攻击者难以通过语法收紧排除蜜罐代码。
