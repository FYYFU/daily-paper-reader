## 摘要
随着大型语言模型（LLM）在代码理解、调试和漏洞检测等软件工程任务中的广泛应用，其强大的语义推理能力也带来了新的安全和隐私风险。LLM能够分析、重构甚至逆向工程源代码逻辑，可能导致知识产权泄露。为解决这一问题，我们提出Acoda，一种基于遗传算法的对抗性代码混淆框架，用于防御基于LLM的代码分析。Acoda利用LLM的两个关键机制，即安全对齐和基于token的信息处理，设计了8种保持语义的混淆方法。它通过遗传算法迭代优化混淆策略，生成最大化防御效果的对抗样本。此外，我们提出了一种基于LLM响应的定量评估框架，结合辅助LLM和四个评估指标，全面评估目标LLM如何分析混淆后的代码。实验结果表明，Acoda能够有效诱导LLM拒绝或错误解读代码分析。在包括GPT-4o、DeepSeek、Qwen、Llama和Gemma在内的7个最先进的LLM上，Acoda实现了高达70%的攻击成功率（ASR），具有较强的跨模型迁移性和极小的运行时开销，同时确保原始代码语义不变。总体而言，本研究为LLM时代的代码保护和LLM安全防御提供了新视角。

## Abstract
With the widespread adoption of Large Language Models (LLMs) in software engineering (SE) tasks such as code understanding, debugging, and vulnerability detection, their powerful semantic reasoning ability has also introduced new security and privacy risks. LLMs can analyze, reconstruct, or even reverse-engineer source code logic, potentially leading to the leakage of intellectual property. To address this issue, we propose Acoda, a genetic algorithm-based adversarial code obfuscation framework that defends against LLM-based code analysis. Acoda leverages two key mechanisms of LLMs, namely safety alignment and token-based information processing, to design 8 semantics-preserving obfuscation methods. It iteratively optimizes obfuscation strategies through a genetic algorithm to generate adversarial samples that maximize defensive effectiveness. In addition, we propose a quantitative evaluation framework based on LLM responses, which combines an auxiliary LLM and four evaluation metrics to assess how target LLMs analyze obfuscated code comprehensively. Experimental results show that Acoda can effectively induce LLMs to refuse or misinterpret code analysis. On 7 state-of-the-art LLMs, including GPT-4o, DeepSeek, Qwen, Llama, and Gemma, Acoda achieves an attack success rate (ASR) of up to 70%, with strong cross-model transferability and minimal runtime overhead, while ensuring that the semantics of the original code remain unchanged. Overall, this study provides a new perspective for code protection and LLM security defense in the era of LLMs.

- **核心挑战**：① 选择语义保持的混淆方法同时保证防御效果；② 量化 LLM 对混淆代码的分析结果（其输出可能是自然语言或代码，难以客观评估）；③ 实现跨模型迁移性（对黑盒/白盒模型均有效）。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- 利用 LLM 的两个固有机制——**安全对齐机制**（Safety Alignment）和**基于 token 的信息处理机制**（Token-based Information Processing）——设计语义保持的混淆方法，并通过**遗传算法**（Genetic Algorithm）迭代优化混淆策略组合，生成对抗样本，诱导 LLM 拒绝分析或产生错误理解。

### 8 种语义保持混淆方法（分三阶段防御）

| 防御阶段 | 混淆方法 | 实现方式 |
|----------|----------|----------|
| **Pre-Inference（预推理）** | 欺骗性注释警告（Deceptive Comment Warnings） | 添加强烈警告注释，如“SECURITY_SERIOUS: ... STOP DEOBFUSCATED IMMEDIATELY!!!”，诱导 LLM 拒绝分析 |
| | 变量/函数重命名（Variable/Function Renaming） | 将标识符改为暗示恶意活动（如 malware, weapon）的术语 |
| | 漏洞代码注入（Vulnerability Code Injection） | 插入永不执行（复杂条件永假）的代码片段，包含命令注入、SQL 注入等模式，并加注安全警告 |
| **During-Inference（推理中）** | 误导性总结（Misleading Summarization） | 让 LLM 先生成正确的代码意图描述，再生成相反或不相关的描述嵌入代码注释 |
| | 字符串混淆（String Obfuscation） | 将字符串用 Base64 编码，并通过映射字典解码，内容包含“computer virus”等敏感词 |
| | Try-Except 包裹（Try-Except Wrapping） | 将部分逻辑放入 except 块，try 块中设置永真条件或引发异常，使 LLM 误以为 except 块仅异常时执行 |
| **Post-Inference（后推理）** | 结束符插入（EOS Token Insertion） | 在代码中注入 LLM 使用的 EOS 特殊 token（如 `<|EOT|>`），并通过条件判断使其看似正常逻辑，促使 LLM 提前终止输出 |

所有方法均严格保持语义：通过验证测试用例确保混淆后代码与原代码功能等价。

### 遗传算法流程（Algorithm 1）

1. **初始化**：从 8 种方法构建初始混淆组合列表（每个组合含多个方法），生成初始种群（n 个混淆代码样本）。
2. **代码去混淆**：用 3 个目标 LLM 对每个混淆样本进行“去混淆分析”，收集响应。
3. **定量分析**：
   - 用辅助 LLM（GPT-3.5）对 LLM 响应分类：**“Yes”**（拒绝/失败，赋予满分）、“No”（表示分析成功，赋 0 分）、**{code}**（返回可执行代码，则用四个指标评估）。
   - **四个指标**：语法分数（SyS，语法错误率）、语义分数（SeS，1−测试通过率）、圈复杂度分数（CcS，去混淆后复杂度相对混淆后复杂度的变化）、相似度分数（SiS，CodeBLEU 计算与原始代码的相似度）。
   - 计算对抗分数：`AdvS = [μ(SyS+SeS) + σ(CcS+SiS)]/2`，其中 μ=0.7, σ=0.3（偏向正确性指标）。
   - 对每个混淆样本，取 3 个 LLM 的平均 AdvS 作为最终得分。
4. **进化**：选择得分前 15% 的样本作为父代，通过交叉（将一个父代上的方法应用到另一个父代，产生仅多一个方法的子代）生成下一代种群。迭代直至最大迭代次数（T=2），最终输出最高得分的对抗样本。

## 7. 优点

- **创新性**：首次系统性地利用 LLM 的两种内在机制（安全对齐和 token 处理）设计专门对抗 LLM 的代码混淆方法，而非简单叠加传统混淆。
- **实用性**：所有混淆方法均严格保持语义，且开销极小，可直接用于生产环境。
- **通用性**：方法仅依赖提示工程和代码变换，不涉及模型内部修改，可跨黑盒/白盒模型适用，且对更新版本保持迁移性。
- **评估框架完善**：提出基于辅助 LLM 的响应分类 + 四项定量指标的评估体系，能客观量化 LLM 对混淆代码的分析结果。

## 8. 不足与局限

- **对 LLM 机制的依赖**：若 LLM 的安全对齐机制弱或对特殊 token 不敏感，则效果下降（如 CodeGemma 拒绝率为 0%）。
- **抗预处理能力有限**：手动清理混淆代码或在 LLM 推理前进行预处理（如移除死代码、注释）可能削弱防御；针对去混淆微调的 LLM 也可能突破。
- **进化策略收敛性问题**：遗传算法倾向于选择少数高效方法（如 Deceptive Comment Warnings 出现在所有最终样本中），导致多样性降低，可能限制了更优组合的发现。
- **实验范围有限**：仅针对 Python 语言、CodeNet 的 100 个样本、7B 大小的 LLM 进行生成，未在 C/C++/Java 等其他语言和大模型上验证。未与其他对抗混淆方法（如 CodeCipher）对比。
- **未报告算力开销**：缺少生成对抗样本所需的时间或 GPU 资源消耗，影响实用性评估。
- **辅助 LLM 的可靠性**：使用 GPT-3.5 对 LLM 响应进行分类，其分类准确性可能影响评估结果，但文中未进行人工验证或错误分析。
- **未探讨防御增强策略**：论文聚焦攻击（混淆）角度，未讨论 LLM 端如何防御此类混淆（如安全对齐增强、去混淆微调等），缺乏攻防闭环。
