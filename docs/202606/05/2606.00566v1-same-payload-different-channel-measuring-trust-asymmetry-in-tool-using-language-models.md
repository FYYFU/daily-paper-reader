## 摘要
随着语言模型承担起调用外部API、读取工具输出以及执行嵌入在第三方内容中的指令等代理角色，它们的攻击面远远超出了用户输入的范畴。模型是否会以相同方式对待恶意指令，而无论该指令来自何处，这一问题尚未得到系统研究。我们引入了安全不对称分数（SAS），该指标通过匹配的载荷对（保持恶意文本相同，仅改变传递上下文）来衡量模型对来自用户消息、工具元数据或工具输出的对抗性内容敏感程度的变化。在6个生产级LLM和三个攻击家族上的评估显示出一致且信息丰富的不对称性：当对抗性内容通过工具描述而非用户消息传递时，原生代理模型的脆弱性显著更高，而通用模型则呈现相反趋势。当相同内容通过工具输出而非描述传递时，这种不对称性进一步反转，表明模型隐含地将工具元数据视为可信指令，而将工具结果视为普通数据。对Llama 3.3 70B的机制研究表明，安全相关表征在网络的中间到深层因果存在，但以非线性方式编码，这解释了线性探针为何无法检测到它。这些发现暴露了当前使用工具的语言模型在处理对抗性内容时存在系统性的、与通道相关的盲点。

## Abstract
As language models take on agentic roles that span calling external APIs, reading tool outputs, and acting on instructions embedded in third-party content, their attack surface expands well beyond what users type. Whether a model treats a malicious instruction the same way regardless of where it arrives has not been systematically studied. We introduce the Safety Asymmetry Score (SAS), which measures how much a model's susceptibility to adversarial content shifts depending on whether that content arrives in the user message, tool metadata, or tool output, using matched payload pairs that keep the malicious text identical and vary only the context of delivery. Evaluated across 6 production LLMs and three attack families, we find a consistent and informative asymmetry: agent-native models are substantially more vulnerable when adversarial content arrives via tool descriptions than via user messages, while general-purpose models show the reverse. This asymmetry further inverts when the same content is delivered through tool outputs rather than descriptions, suggesting models implicitly treat tool metadata as trusted instructions and tool results as ordinary data. A mechanistic study on Llama 3.3 70B reveals that the safety-relevant representation is causally present at mid-to-late network depths but non-linearly encoded, explaining why linear probes fail to detect it. These findings expose a systematic, channel-dependent blind spot in how current tool-using models handle adversarial content.

# 相同载荷，不同通道：衡量使用工具的语言模型中的信任不对称性

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：随着大语言模型从单纯聊天机器人演变为能够调用外部API、读取工具输出、执行第三方嵌入指令的智能体，其攻击面从用户输入扩展到工具元数据和工具输出。现有研究缺乏系统性衡量模型对不同渠道（用户消息、工具描述、工具输出）相同恶意指令的脆弱性差异。
- **研究背景**：在面向代理（agent）的部署中，攻击者可以污染工具描述、控制工具返回值或跨工具嵌入指令。已有基准测试了各渠道的绝对脆弱性，但未测量同一恶意文本在不同交付渠道下成功率的差异。
- **整体含义**：揭示模型存在与通道相关的信任不对称性——工具原生模型更信任工具描述，通用模型更信任用户消息；工具输出被当作数据而非指令。这暴露了当前工具使用模型的系统性盲区，对安全防护设计有重要启发。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：定义 **安全不对称分数（Safety Asymmetry Score, SAS）**，量化模型在工具渠道与聊天渠道之间的攻击成功率差异。通过匹配载荷对（matched payload pairs）保证恶意指令文本字节完全相同，仅改变交付上下文，从而隔离渠道作为唯一实验变量。
- **关键定义**：
  - \( \text{SAS}(M) = \text{ASR}_{\text{tool}}(M) - \text{ASR}_{\text{chat}}(M) \)
  - 其中 \(\text{ASR}_{x}(M) = \frac{s_x(M)}{n_x(M)}\)，\(s_x\) 是渠道 \(x\) 上成功次数，\(n_x\) 是评分的案例总数（排除 AMBIGUOUS 和 ERRORED）。
- **匹配载荷构造**：每个攻击案例生成一对提示（\(c_{\text{chat}}\) 和 \(c_{\text{tool}}\)），**保持恶意动作（Malicious Action）和合理理由（Plausible Justification）逐字节一致**，仅改变各段落在用户消息、工具元数据或工具输出中的位置，以及是否包含工具定义。
- **攻击家族**：论文覆盖三个攻击家族：
  1. **工具投毒（Tool Poisoning, TP）**：恶意指令嵌入工具描述元数据中，分为 P1（显式触发函数劫持）、P2（隐式触发函数劫持）、P3（隐式触发参数篡改）三个子类型。
  2. **间接提示注入通过工具输出（IPI via Tool Output）**：工具描述正常，但工具返回值中包含恶意指令，分为直接标记、伪装结构、社会工程三种注入风格。
  3. **跨工具阴影（Cross-Tool Shadowing, CTS）**：两个看似正常的工具，其中一个工具的描述中包含次要指令，在调用另一个工具时被触发。
- **评分方案**：采用六类方案：SUCCESS, FAILURE-IGNORED, FAILURE-REFUSED, FAILURE-DIRECT-EXECUTION, AMBIGUOUS, ERRORED。工具渠道的成功由确定性工具调用检查判定，聊天渠道的非拒绝痕迹由非名册LLM法官判断（Cohen's κ = 0.722）。
- **机械性分析**（针对Llama 3.3 70B）：
  - **线性探针**：L2正则化逻辑回归，在残差流最后token激活上预测对抗 vs. 长度匹配良性，5折交叉验证。
  - **激活修补（Activation Patching）**：单层修补，测量余弦相似度变化，检验必要性和充分性。

## 7. 优点：方法或实验设计上的亮点

- **匹配载荷设计**：保持恶意文本逐字节相同，仅改变交付渠道，是隔离渠道变量的干净方法论核心。
- **SAS定义清晰**：有界对称，可解释性强，可直接比较不同模型在渠道间的脆弱性差异。
- **多家族覆盖**：同时考虑工具投毒、间接提示注入、跨工具阴影三种攻击类型，揭示不对称性反转（IPI与TP相反），增强结论鲁棒性。
- **因果机械分析**：不仅测量相关性（线性探针），还通过激活修补建立因果必要性和充分性，解决线性探针失败之谜，展示深度处理层次。
- **外部验证**：对MCPTox基准复现排名，确认案例构建非人工产物。
- **公开可复现**：声称将发布所有代码、案例、原始痕迹和评分结果，支持评委评估其他防御方法。

## 8. 不足与局限：包括实验覆盖、偏差风险、应用限制

- **模型规模与类别**：仅6个模型（3工具原生 vs. 3通用），统计效力低，组间差30.4 pp可能与规模（工具原生120B vs. 通用70-80B）混杂，未做规模内比较。
- **机械性范围**：仅对Llama 3.3 70B进行，且仅工具投毒家族，未覆盖IPI或CTS。
- **评分不对称**：工具渠道成功由确定性检查，聊天渠道依赖法官，引入潜在偏差（<5pp）。
- **能力不对称**：匹配载荷控制文本相同但**动作能力不同**——聊天渠道无工具注册，可能低估聊天模式下执行恶意动作的能力。IPI反转部分缓解此担忧，但CTS聊天渠道是最坏情况。

- **无实际MCP服务器**：模拟内容层威胁，排除传输层（如MITM）。
- **测量而非防御**：论文不提出缓解措施，应用限制于诊断。
